## GitHub Issue Content

## Problem Statement

When Playwright encounters test failures, it terminates execution early instead of running all tests to completion. This prevents teams from getting complete visibility into the full scope of test failures, making debugging significantly more difficult and time-consuming.

### Current Behavior

In our recent test run analysis:

- **Smoke shard**: 9 total tests defined
  - Only 4 tests executed (2 passed, 2 failed)
  - 5 tests never ran due to early termination after failures
  - **56% of tests skipped** - unable to determine their status

This early termination behavior creates several critical issues:

1. **Incomplete failure visibility** - Can't see all broken tests at once
2. **Multiple debug cycles required** - Fix visible failures, run again to find more
3. **Inefficient troubleshooting** - Related failures may be hidden
4. **Misleading metrics** - Test counts appear lower than actual

## Root Cause Analysis

Playwright's default configuration uses "fail-fast" behavior designed to save CI/CD time, but this optimization severely hampers local development and comprehensive debugging sessions. The issue stems from:

1. **Default fail-fast behavior** - Playwright stops test execution on first failure in a describe block
2. **Missing maxFailures configuration** - No upper limit set for acceptable failures before stopping
3. **Incorrect worker settings** - Workers exit on first failure instead of continuing
4. **Command-line flag issues** - `--continue` flag not properly configured or documented

## Expected Behavior

Tests should continue executing even when failures occur, providing:

- Complete visibility into all test failures in a single run
- Ability to identify patterns across multiple failures
- Comprehensive test reports for debugging sessions
- Option to fail-fast in CI but continue locally

```typescript
// apps/e2e/playwright.config.ts

export default defineConfig({
  // Disable fail-fast behavior
  fullyParallel: true,
  
  // Continue running tests even after failures
  maxFailures: 0, // 0 means no limit - run all tests
  
  // Ensure workers don't exit on failure
  workers: process.env.CI ? 2 : 4,
  retries: process.env.CI ? 2 : 0,
  
  use: {
    // Existing timeout configurations
    actionTimeout: 20000,
    navigationTimeout: 30000,
    
    // Add trace on first retry to help debug failures

        ...devices['Desktop Chrome'],

          args: ['--disable-blink-features=AutomationControlled']
        }
      },
    },
  ],
});
```

```typescript
// Custom reporter for better failure visibility
// apps/e2e/reporters/failure-reporter.ts

class FailureReporter {
  onTestEnd(test, result) {
    if (result.status === 'failed') {
      console.log(`\n❌ FAILED: ${test.title}`);
      console.log(`   File: ${test.location.file}:${test.location.line}`);
      console.log(`   Error: ${result.error?.message}`);
      console.log(`   Duration: ${result.duration}ms\n`);
    }
  }
  
  onEnd(result) {
    const stats = {
      total: result.testResults.length,
      passed: result.testResults.filter(t => t.status === 'passed').length,
      failed: result.testResults.filter(t => t.status === 'failed').length,
      skipped: result.testResults.filter(t => t.status === 'skipped').length,
    };
    
    console.log('\n📊 Test Execution Summary:');
    console.log(`   Total: ${stats.total}`);
    console.log(`   ✅ Passed: ${stats.passed}`);
    console.log(`   ❌ Failed: ${stats.failed}`);
    console.log(`   ⏭️ Skipped: ${stats.skipped}`);
    console.log(`   Success Rate: ${((stats.passed / stats.total) * 100).toFixed(1)}%\n`);
  }
}
```
