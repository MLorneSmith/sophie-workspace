## 🐛 Bug Description

VS Code is not properly connecting to GitHub Codespaces when launched from WSL. Multiple issues are preventing the intended workflow of having separate VS Code windows each connected to different codespaces.

## Current Issues

### 1. Terminal Opens to Local Directory

- **Expected**: Terminal should connect to remote codespace (e.g., `root@codespace:/workspace$`)
- **Actual**: Terminal opens to local project folder `➜  2025slideheroes git:(dev) ✗`

### 2. VS Code Tab Shows Raw Connection String

- **Expected**: Tab should show codespace name or project name
- **Actual**: Tab displays: `connect?name=animated-adventure-r4j5g7r6496f5qxx&window=_blank`

### 3. Biome Extension Error

- **Error Message**: "Please install Biome globally on your system so the extension can start a global LSP session for files that are not part of any workspace"
- This suggests VS Code thinks it's in a local context rather than remote

### 4. Connection Method Failures

Multiple connection attempts have failed:

- SSH Remote connection: "Could not resolve hostname" error
- GitHub Codespaces URL: Opens locally instead of remote
- GitHub CLI (`gh codespace code`): "xdg-open,x-www-browser,www-browser,wslview" not found

## Root Cause Analysis

The core issue appears to be that VS Code Insiders from WSL cannot properly invoke the GitHub Codespaces connection. The URL scheme is being interpreted as a file path rather than a protocol handler.
