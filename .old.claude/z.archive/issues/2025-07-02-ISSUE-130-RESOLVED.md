### Root Cause

The pptxgenjs library is a Node.js-only library that uses file system operations (fs, fs/promises) and https module. The library was being imported in client-side React components with the "use client" directive, causing webpack to attempt bundling Node.js modules for browser use, which is not possible.

The import chain was:

- `slide-preview.tsx` (client component) → imported `LAYOUT_POSITIONS` from `pptx-generator.ts`
- `powerpoint-exporter.tsx` (client component) → imported `PptxGenerator` class from `pptx-generator.ts`
- `pptx-generator.ts` → imported `pptxgenjs` which requires Node.js modules

### Solution Implemented

1. **Created a constants file** (`layout-positions.ts`) to hold the `LAYOUT_POSITIONS` constant that can be safely imported by both client and server components

2. **Created a server action** (`powerpoint-actions.ts`) that wraps the PowerPoint generation functionality, ensuring it only runs on the server where Node.js modules are available

3. **Updated client components**:
   - `slide-preview.tsx` now imports `LAYOUT_POSITIONS` from the constants file
   - `powerpoint-exporter.tsx` now calls the server action instead of directly using `PptxGenerator`

4. **Updated `pptx-generator.ts`** to import and re-export from the constants file for backward compatibility

### Lessons Learned

1. **Server-only libraries must be isolated**: Libraries that use Node.js modules must be wrapped in server actions or API routes when needed by client components
2. **Constants should be extracted**: Shared constants between client and server code should be in separate files to avoid unnecessary dependencies
3. **Import chains matter**: Even importing a single constant from a file that imports Node.js modules will cause bundling issues in client components
4. **Next.js server actions are the solution**: They provide a clean way to execute server-only code from client components
