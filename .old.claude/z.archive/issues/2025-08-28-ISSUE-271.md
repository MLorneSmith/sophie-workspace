## Problem Statement

The test-controller.cjs script used by the /newtest slash command is critically underreporting unit test counts, showing only ~60 tests when the project actually contains **498 unit tests**. This creates a false impression of test coverage and masks the true scope of the test suite.

## Investigation Findings

### Actual Test Distribution (498 Total Unit Tests)

```
📊 Unit Test Distribution:
├── apps/web: 434 tests (17 test files)
│   ├── components: 406 tests

│   └── lib/trpc: 4 tests
├── apps/payload: 60 tests (2 test files)
│   ├── collections: 58 tests
│   └── utilities: 2 tests
└── packages: 4 tests (2 test files)
    ├── @kit/monitoring-newrelic: 1 test
    └── @kit/shared: 3 tests

Total: 498 unit tests across 21 test files
```

### Root Cause Analysis

The test-controller.cjs script relies on `pnpm test:unit` which executes:

```bash
turbo test --cache-dir=.turbo --filter=!web-e2e
```

**Critical Issue**: Turbo's aggressive caching behavior means:

1. **Cache Hit Scenario**: If no code has changed in a workspace, Turbo assumes tests don't need to run
2. **Skipped Execution**: The main web app with 434 tests gets completely skipped when cached
3. **False Reporting**: The controller only sees and reports tests that actually execute
4. **Misleading Metrics**: Only 60 tests from payload and 1 from monitoring run, while 437 are silently skipped

### Evidence from Test Runs

When running with cached results:

```
✓ @kit/monitoring-newrelic (1 passed, 4 skipped)
✓ payload (60 passed)
❌ web (SKIPPED - 434 tests not executed due to cache)
```

The controller sees this as "61 tests passed" when in reality 437 additional tests exist but weren't executed.

// Add test discovery phase before execution
const discoverTests = async () => {

    ignore: ['node_modules/**', 'dist/**', '.next/**']
  });
  
  // Parse files to count test blocks

  return testCounts;
