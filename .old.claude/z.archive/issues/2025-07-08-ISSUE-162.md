## Summary

The CI/CD pipeline is running duplicate security scans (TruffleHog and Semgrep) when promoting code from dev to staging via pull request. These scans were just completed when pushing to the dev branch, and running them again immediately for the same code is inefficient and wastes CI resources.

## Environment

- **Application Version**: Current dev branch
- **Environment**: CI/CD Pipeline (GitHub Actions)
- **Last Working**: N/A - Design inefficiency
- **Recent Commits**: 18f86bbe (dev branch)

## Current Behavior

1. Push to `dev` branch triggers:
   - TruffleHog Secret Scan
   - Semgrep Security Scan
   - Other validation checks

2. Creating PR from `dev` to `staging` immediately triggers:
   - TruffleHog Secret Scan (again)
   - Semgrep Security Scan (again)
   - Same validation checks

## Expected Behavior

According to `.claude/context/systems/cicd-pipeline-design.md`:

- **Phase 2 (PR Validation)**: Run security scans on feature → dev PRs
- **Phase 3 (Dev Deploy)**: Skip redundant scans, focus on deployment
- **Phase 4 (Staging)**: Run staging-specific tests (E2E, performance), not duplicate security scans

The pipeline should intelligently skip security scans that have already passed for the same commits.

## Initial Analysis

The workflows are not checking whether the commits being promoted have already been scanned. This is particularly inefficient for branch-to-branch promotions where no new code has been introduced.

## Suggested Investigation Areas

1. **Implement commit-based caching**: Check if commits have already been scanned
2. **Use conditional workflow logic**: Skip scans for branch promotions
3. **Add workflow dispatch inputs**: Allow manual override when needed

## Proposed Solutions

### Option 1: Conditional Execution Based on Source Branch

```yaml
security-scan:
  if: github.event.pull_request.head.ref != 'dev'
  # Only run if not promoting from dev
```

### Option 2: Commit Hash Checking

```yaml
- name: Check if commits already scanned
  id: scan-check
  run: |
    # Check if these commits passed security scans in another workflow
    # Use GitHub API to check previous workflow runs
```

### Option 3: Workflow Optimization

- Move security scans to only run on:
  - Direct pushes to protected branches
  - PRs from feature branches
  - Manual workflow dispatch
