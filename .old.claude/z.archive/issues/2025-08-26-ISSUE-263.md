## Description

Our test infrastructure exhibits flaky behavior where tests fail on first attempt but pass on retry, indicating infrastructure instability rather than actual test failures. This is causing unreliable CI/CD pipelines and reduced developer confidence.

## Current State

- Tests randomly fail on first attempt but pass on retry
- Infrastructure failures occur sporadically across different shards
- Retry logic masks underlying infrastructure problems  
- Affects both unit and E2E tests, but more prevalent in E2E

## Symptoms Observed

1. **Port conflicts**: Tests fail due to ports already in use
2. **Database connection issues**: Intermittent Supabase connection failures
3. **Process cleanup failures**: Zombie processes from previous test runs
4. **Resource contention**: Tests interfere with each other during parallel execution
5. **State leakage**: Tests not properly isolated, causing cascading failures

## Specific Issues

- Playwright processes not being killed properly between runs
- Next.js dev server instances lingering after test completion
- Supabase connections not being closed cleanly
- Port 3000, 3020, and 55321-55322 sometimes remain occupied
- Test database not properly reset between test suites

## Impact

- CI/CD pipeline success rate below 70% (target: 95%+)
- Developers lose 30-45 minutes daily to re-running failed tests
- False-positive failures hide real issues
- Reduced confidence in test suite reliability
- Increased infrastructure costs from repeated test runs

## Root Causes

1. Insufficient process cleanup between test runs
2. Lack of proper test isolation
3. Missing health checks before test execution
4. No resource locking for shared resources
5. Inadequate error handling in setup/teardown

## Suggested Solutions

### 1. Enhanced Process Cleanup

```bash
# Kill all test-related processes
pkill -f "playwright|vitest|next-server" || true
lsof -ti:3000-3020,55321-55322 | xargs kill -9 2>/dev/null || true
```

### 2. Test Isolation

- Use unique database schemas per test suite
- Implement resource locking for ports
- Add test namespacing for parallel execution

### 3. Pre-flight Health Checks

- Verify all ports are free before starting
- Check database connectivity
- Ensure clean process state
- Validate environment configuration

### 4. Robust Setup/Teardown

- Implement guaranteed cleanup in finally blocks
- Add timeout handling for all async operations
- Use process groups for better process management

## Implementation Plan

1. Add comprehensive pre-flight checks in test-controller.cjs
2. Implement process group management for test processes
3. Create test isolation framework
4. Add resource locking mechanism
5. Implement health check endpoints
