## Summary

E2E accessibility tests are failing due to Tailwind CSS v4 generating incorrect color values from CSS custom properties. Despite defining CSS variables with colors that meet WCAG AA contrast requirements (4.5:1), the generated CSS classes (text-primary, text-foreground, etc.) are producing colors with insufficient contrast ratios (2.07:1 - 4.17:1). This appears to be a fundamental issue with how Tailwind CSS v4 processes color variables.

- **Browser**: Chromium (Playwright)

## Reproduction Steps

1. Define CSS variables in `shadcn-ui.css` with high-contrast colors
2. Run E2E accessibility tests: `cd apps/e2e && pnpm test -- --grep "accessibility"`
3. Tests fail with color contrast violations
4. Browser DevTools show different computed colors than expected

## Expected Behavior

When defining CSS variables with appropriate color values:

- `--foreground: 0 0% 15%` (HSL) or `#262626` (hex) should produce 15.1:1 contrast
- `--muted-foreground: 0 0% 32%` (HSL) or `#525252` (hex) should produce 7.8:1 contrast
- These values should be used by Tailwind's `text-foreground` and `text-muted-foreground` classes

## Actual Behavior

Tailwind CSS v4 is generating incorrect colors:

- `text-foreground` elements showing `#909090` (3.19:1 contrast) instead of expected `#262626`
- `text-primary` elements showing `#b4b4b4` (2.07:1 contrast) instead of expected values
- Colors change with each attempted fix, suggesting CSS processing/caching issues
