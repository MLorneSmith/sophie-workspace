## Description

E2E tests are experiencing critical WebServer timeout issues during sharded execution. Some shards fail while waiting for the test servers (Next.js frontend and Payload backend) to start, leading to test failures and infrastructure instability.

## Current State

- Some shards timeout waiting for server startup
- Retries sometimes succeed, indicating timing/race conditions  
- Affects approximately 20-30% of E2E test runs
- More prevalent when running multiple shards concurrently

## Error Patterns Observed

- `WebServer Timed out` errors in test logs
- `ECONNREFUSED` when tests try to connect before servers are ready
- Early shard exits (under 2 seconds) with exit code 1
- Inconsistent behavior - same shard may pass on retry

## Root Causes

1. Servers take longer to start under heavy system load
2. Multiple shards competing for resources during parallel execution
3. Current 90-second webServer timeout may be insufficient for cold starts
4. Race condition between server startup and test execution

## Impact

- **Reliability**: Flaky E2E tests reducing confidence in CI/CD pipeline
- **Productivity**: False-positive failures requiring manual re-runs
- **Performance**: Increased test execution time due to retries
- **Developer Experience**: Frustration with unreliable test infrastructure

## Reproduction Steps

1. Run E2E tests with multiple shards: `npm run test:e2e`
2. Observe server startup logs and timing
3. Note failures with "WebServer Timed out" errors
4. Re-run failed shards - some will pass on retry

## Environment

- **Test Framework**: Playwright

## Suggested Fixes

### 1. Immediate (Hours)

Increase webServer timeout from 90s to 120s in `playwright.config.ts`:

```javascript
webServer: {
  timeout: 120 * 1000, // Increase from 90s
  reuseExistingServer: true,
}
```

### 2. Short-term (Days)

Pre-warm servers before shard execution:

```javascript
// In test-controller.cjs
async startTestServers() {
  await this.startServersWithRetry();
  await this.waitForHealthCheck();
  // Only then allow shards to proceed
}
```

### 3. Medium-term (Week)

Implement proper health check endpoints:

```javascript
// Add health endpoints
GET /api/health -> { status: 'ready', services: {...} }
GET /admin/health -> { status: 'ready', db: 'connected' }

// Poll for ready state
async waitForServersReady() {
  const maxRetries = 60;
  for (let i = 0; i < maxRetries; i++) {
    if (await this.checkHealth()) return true;
    await sleep(2000);
  }
  throw new Error('Servers failed to become ready');
}
```

### 4. Long-term (Sprint)

Consider persistent test server instances:

- Keep test servers running between test runs
- Reset data/state instead of full restart
- Use Docker containers for isolation
