# Review Implementation

Review implemented work against a specification from a GitHub issue to ensure it matches requirements. Capture visual evidence of functionality and create detailed review reports with issue tracking.

## Instructions

IMPORTANT: You're reviewing implemented work against a specification to ensure it matches requirements.
IMPORTANT: This is about validation and quality assurance, not implementation or testing.
IMPORTANT: Focus on verifying that implemented features match the spec for critical functionality.
You're creating a comprehensive review report with visual evidence and issue tracking.

### 1. Fetch Implementation Issue from GitHub

**Fetch the implementation issue**:
```bash
# Extract issue number from $ARGUMENTS and fetch the issue with comments
gh issue view <issue-number> \
  --repo slideheroes/2025slideheroes \
  --json body,title,labels,number,url,comments \
  --jq '{body: .body, title: .title, labels: [.labels[].name], number: .number, url: .url, comments: [.comments[] | {author: .author.login, body: .body, createdAt: .createdAt}]}'
```

**Validate it's reviewable**:
- Check for "implemented" label OR "ready-for-review" label
- If not ready for review, inform user the issue needs to be implemented first
- Extract the spec file path from the issue body (look for file paths in .ai/specs/)

### 8. Conduct Review

**Compare implementation against spec**:
- Read the spec file carefully to understand all requirements
- Review git changes to see what was implemented
- Identify all critical functionality described in the spec
- Verify each critical feature was implemented as specified

**If UI validation is needed** (user-facing features):
- Look for corresponding e2e test files in `apps/e2e/tests/**/*.spec.ts` that mirror the feature
- Use e2e test files ONLY as navigation guides for screenshot locations
- **IMPORTANT**: We're reviewing, not testing. We know the functionality works.
- **IMPORTANT**: Take screenshots to showcase the new functionality and any issues found

**Screenshot Guidelines**:
- Capture visual proof of working features through targeted screenshots
- Navigate to critical paths based on the spec requirements
- Compare implemented UI/functionality with spec requirements
- **Aim for 1-5 screenshots** to showcase critical functionality
- Number screenshots: `01_<descriptive-name>.png`, `02_<descriptive-name>.png`, etc.
- **IMPORTANT**: Capture screenshots of critical points of new functionality
- **IMPORTANT**: Store screenshots in `reviewImageDir` using full absolute paths
- Focus only on critical functionality paths - avoid unnecessary screenshots
- Use descriptive filenames that indicate what part of the feature is being verified

**If issues found**:
- Take a screenshot of the issue
- Add to `review_issues` array with:
  - Screenshot path
  - Issue description
  - Proposed resolution
  - Severity (skippable, tech_debt, or blocker)

### 9. Assess Issues by Severity

**IMPORTANT**: Think hard about the impact of each issue on the feature and user.

**Severity Guidelines**:
- `skippable` - Non-blocker for release but is still a problem (minor UI issues, non-critical bugs)
- `tech_debt` - Non-blocker for release but will create technical debt (code quality, performance optimization)
- `blocker` - Blocker for release, harms user experience or prevents functionality (critical bugs, broken features)

## Issue Severity Guidelines

**IMPORTANT**: Think hard about the impact of the issue on the feature and the user experience.

### Skippable
- **Definition**: Non-blocker for release but is still a problem
- **Examples**: Minor UI alignment issues, non-critical typos, cosmetic improvements
- **Impact**: Low - Does not affect core functionality or user experience significantly

### Tech Debt
- **Definition**: Non-blocker for release but will create technical debt that should be addressed
- **Examples**: Code quality issues, missing tests, performance optimizations, refactoring opportunities
- **Impact**: Medium - Should be addressed in future to maintain code health

### Blocker
- **Definition**: Blocker for release, harms user experience or prevents functionality
- **Examples**: Critical bugs, broken features, security vulnerabilities, data loss risks
- **Impact**: High - Must be fixed before release, prevents users from completing critical tasks

If review fails with blocking issues, create a follow-up issue:

```bash

**Original Issue**: #${issueNumber}
**Review Report**: \`.ai/specs/review-${issueNumber}.md\`

## Blocking Issues

${BLOCKING_ISSUES}

## Review Summary

3. Request re-review
```
