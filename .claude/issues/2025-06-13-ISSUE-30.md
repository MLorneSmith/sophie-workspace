# Enhancement: Streamline Issue Logging System

## Problem Statement

Currently, our issue logging system creates both GitHub issues and local files, leading to duplication and inefficiency. The system maintains two versions of the same data that can drift out of sync.

## Proposed Solution: GitHub as Single Source of Truth

Implement Option 1 from the analysis:

- GitHub issues become the authoritative source
- Local files are auto-synced mirrors for search/grep functionality
- Eliminate manual duplication while maintaining search benefits

## Current Workflow Issues

1. **Duplication**: `/log-issue` creates both GitHub issue and local file
2. **Sync drift**: Manual updates can cause inconsistencies
3. **Maintenance overhead**: Need to update both locations
4. **Storage inefficiency**: Duplicate data storage

## Proposed New Workflow

### Phase 1: Core Implementation

1. **Modified `/log-issue`**: Creates GitHub issue only by default
2. **Auto-sync service**: Polls/webhooks to sync GitHub issues to local files
3. **Local file format**: Read-only mirrors with metadata linking to GitHub
4. **Updated `/debug-issue`**: Works seamlessly with both sources

### Phase 2: Enhanced Features

1. **Bidirectional sync**: Local edits push to GitHub (optional)
2. **Offline mode**: Queue changes when GitHub unavailable
3. **Search optimization**: Enhanced local indexing
4. **Conflict resolution**: Handle simultaneous edits

## Technical Implementation

### GitHub API Integration

- Use existing `mcp__github__create_issue` for creation
