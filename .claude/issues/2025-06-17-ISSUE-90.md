## Initial Analysis

2. **PATH Resolution Issue**: The command is being executed in a bash context that may not have access to node_modules/.bin in its PATH.

3. **npx vs Direct Call**: The lint script in package.json uses `npx yaml-lint` which should resolve the binary, but lint-staged uses direct `yaml-lint` command.

4. **Shell Context**: The bash -c execution in lint-staged may create a new shell context without proper PATH setup.

### Root Cause

The root cause was a **binary name mismatch**. The `yaml-lint` npm package provides a binary named `yamllint` (without hyphen), but the lint-staged configuration was trying to call `yaml-lint` (with hyphen). This caused the "command not found" error because the shell couldn't find a binary named `yaml-lint` in the PATH.

### Solution Implemented

Updated the package.json configuration in two places:

1. **Line 25**: Changed `"lint:yaml": "npx yaml-lint '**/*.{yml,yaml}'"` to `"lint:yaml": "npx yamllint '**/*.{yml,yaml}'"`

2. **Line 84**: Changed the lint-staged command from `yaml-lint "$0"` to `npx yamllint "$0"` to ensure proper binary resolution:
   ```json
   "**/*.{yaml,yml}": [
     "bash -c 'if [[ \"$0\" != *\"pnpm-lock.yaml\"* ]]; then npx yamllint \"$0\"; fi'"
   ]
   ```

Using `npx` ensures that the binary is resolved correctly even in different shell contexts, preventing future PATH-related issues.

### Files Modified

- package.json - Updated lint:yaml script and lint-staged configuration to use correct binary name

### Lessons Learned

1. Always verify the actual binary name provided by npm packages - it may differ from the package name
2. Using `npx` in lint-staged configurations is more reliable than direct binary calls
3. Check node_modules/.bin/ directory to see actual binary names when debugging command not found errors
