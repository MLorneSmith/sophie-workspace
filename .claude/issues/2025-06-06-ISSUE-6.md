## Initial Analysis

The issue occurs at the Vite transformation step during test execution, specifically in the `vite:import-analysis` plugin. The error suggests that Vite's import analysis is not properly respecting the alias configuration defined in `vitest.config.ts`.

**Key findings:**

1. ✅ Aliases are correctly configured in vitest.config.ts with absolute paths
2. ✅ Target files exist at the specified locations
3. ✅ Import format matches what works in the actual Next.js application
4. ❌ Vite's import analysis plugin fails to resolve the alias during test execution

**Potential causes:**

- Vitest/Vite version compatibility issue (Vite 6.3.5 with Vitest 3.1.2)
- Alias resolution order/precedence issues
- Module resolution differences between Vitest and Next.js environments
- Vite's import analysis happening before alias resolution

## Resolution

**Issue Resolved**: 2025-06-06T16:32:00Z  
**Root Cause**: Vitest was not properly resolving TypeScript path mappings from tsconfig.json  
**Solution**: Installed and configured `vite-tsconfig-paths` plugin

### Changes Made

1. **Installed `vite-tsconfig-paths` plugin**:

   ```bash
   pnpm add -D vite-tsconfig-paths
   ```

2. **Updated `vitest.config.ts`**:

   ```typescript
   import tsconfigPaths from 'vite-tsconfig-paths';

   export default defineConfig({
     plugins: [
       tsconfigPaths(), // Synchronize TypeScript paths with Vitest/Vite module resolution
     ],
     // ... rest of config
   });
   ```

3. **Enhanced `tsconfig.json` path mappings**:

   - Added specific export paths for packages with subpath exports
   - Configured mappings for `@kit/supabase/server-client`, `@kit/next/actions`, etc.

4. **Added server-side mocking support**:
   - Added `server-only` mock in `vitest.setup.ts`
   - Added mocks for Next.js server functions (`next/cache`, `next/headers`)
