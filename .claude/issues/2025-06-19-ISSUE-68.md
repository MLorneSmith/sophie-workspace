## Summary

When Node.js packages use `"type": "module"` (ESM mode) and attempt to directly import TypeScript files from workspace packages, Node.js throws `ERR_UNKNOWN_FILE_EXTENSION` errors because it cannot process .ts files without a transpilation step. This affects development workflows and CLI tools that need to import TypeScript modules directly.

- **Application Version**: 2.11.0
- **Environment**: development
- **Node Version**: v22.16.0
- **TypeScript Version**: 5.8.3
- **Package Manager**: pnpm
- **Workspace Setup**: Monorepo with mixed ESM/CommonJS packages
- **Last Working**: Not applicable (architectural limitation)

## Reproduction Steps

1. Navigate to any ESM package (e.g., `apps/payload` with `"type": "module"`)
2. Attempt to directly import a TypeScript file from a workspace package:
   ```bash
   node -e "import('@kit/shared/logger').then(console.log)"
   ```
3. Observe the error: `TypeError [ERR_UNKNOWN_FILE_EXTENSION]: Unknown file extension ".ts"`

## Expected Behavior

TypeScript files should be importable in ESM contexts through proper build pipeline or runtime transpilation configuration, allowing seamless integration between workspace packages.

## Initial Analysis

The issue stems from a fundamental incompatibility between:

1. **ESM Package Requirements**: Node.js ESM mode expects JavaScript files or requires explicit loaders for other file types
2. **Workspace Export Strategy**: Current packages export TypeScript source files directly for build-time optimization
3. **Runtime Import Context**: CLI tools and direct Node.js execution cannot handle .ts extensions without transpilation

This is a **build system architecture issue** that affects:

- Development tooling that runs in Node.js context
- CLI commands and scripts
- Test utilities that import workspace packages
- Any runtime environment requiring direct imports

## Suggested Investigation Areas

1. **Export Strategy Review**: Update package.json exports to point to built JavaScript files instead of TypeScript source
2. **Build Pipeline Enhancement**: Ensure all workspace packages build to JavaScript with proper ESM output
3. **Development Tooling**: Configure development tools to handle TypeScript imports (tsx, ts-node/esm, etc.)
4. **Monorepo Configuration**: Standardize build/export patterns across all workspace packages
5. **Runtime Compatibility**: Test import resolution across different runtime contexts

## Potential Solutions

### Option 1: Build-First Approach
