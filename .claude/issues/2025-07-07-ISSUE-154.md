# Issue: Pre-commit hooks suppress errors allowing invalid code to be committed

## Summary

The pre-commit hooks are configured to suppress all linting and type checking errors using `|| true`, which allows code with syntax errors and type violations to be committed. Additionally, the TypeScript check has a short timeout and is non-blocking. This configuration allowed the syntax errors in issue #153 to be committed and only caught them when deploying to staging.

## Initial Analysis

The pre-commit hook has two critical flaws:

1. **Error Suppression**: Every linting command ends with `|| true` or redirects errors to `/dev/null`, which means failures are ignored
2. **Weak TypeScript Validation**:
   - 30-second timeout is too short for large codebases
   - Failures only show a warning message
   - Not integrated with the main validation flow

This creates a situation where developers can commit broken code that only fails in CI/CD pipelines, leading to blocked deployments and wasted time.

## Suggested Investigation Areas

1. Remove all `|| true` statements from linting commands
2. Remove `2>/dev/null` to show actual error messages
3. Increase TypeScript timeout or remove it entirely
4. Make TypeScript check a required step that blocks commits on failure
5. Consider adding `--max-warnings 0` to ESLint commands
6. Add explicit syntax checking before other validations

## Additional Context

### Current Hook Flow

2. Formatting runs but errors are suppressed
3. TypeScript check runs with timeout, failures only warn

### Proposed Hook Flow

1. Syntax check (fast, fails on syntax errors)
2. Linting (fails on errors)
3. Type checking (no timeout, fails on errors)
4. Only re-stage files if all checks pass
5. Block commit if any check fails
