## Root Cause

The issue was caused by lint-staged running too many concurrent tasks, leading to resource exhaustion and processes being terminated with SIGKILL. By default, lint-staged runs with infinite concurrency (`concurrent: true`), which can overwhelm system resources when multiple lint/format tasks execute in parallel.

## Solution Implemented

Modified the pre-commit hook to limit lint-staged concurrency using the `--concurrent 2` CLI flag. This restricts lint-staged to running a maximum of 2 tasks in parallel, preventing resource exhaustion.

- `.husky/pre-commit` - Added `--concurrent 2` flag to the `npx lint-staged` command

## Lessons Learned

1. **Resource Management**: When running multiple build/lint tools in parallel, it's important to consider system resource constraints
2. **CLI vs Config**: The `concurrent` option in lint-staged is a CLI parameter, not a configuration file option
3. **Default Behavior**: lint-staged's default infinite concurrency can cause issues in resource-constrained environments (WSL2, Docker, CI)

## Additional Notes

- The concurrency limit of 2 was chosen as a conservative value that should work in most environments
- If SIGKILL errors persist, the value can be reduced to 1 (serial execution) or increased if more resources are available
- This fix is particularly important for WSL2 environments which may have more limited resources
