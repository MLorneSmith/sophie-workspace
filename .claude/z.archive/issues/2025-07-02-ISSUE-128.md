### Additional Research Findings

From Tailwind CSS v4 documentation and shadcn/ui migration guides:

1. **Tailwind v4 Color System Changes**:
   - Colors are now converted to OKLCH internally
   - The `@theme inline` approach is the new way to define theme values
   - CSS variables must be properly wrapped with `hsl()` when used

2. **Common Issues Identified**:
   - CSS variables with HSL not rendering correctly when the variable is not formatted properly
   - The browser needs the full `hsl()` context to interpret the variable as a color
   - Multiple `@theme` blocks might conflict or not process correctly

3. **Configuration Requirements**:
   - Tailwind v4 uses CSS-based configuration (no more `tailwind.config.js`)
   - The `@theme` directive must be processed by Tailwind
   - Import order matters - theme definitions should come after Tailwind import

### What We've Tried That Didn't Work

1. **Inline styles with hex**: `style={{ color: "#262626" }}` - Still computed as `#9e9e9e`
2. **Inline styles with HSL**: `style={{ color: "hsl(0 0% 15%)" }}` - Still computed as `#979797`
3. **Using `text-foreground` class**: Produces incorrect color
4. **Changing import order**: Moving `shadcn-ui.css` after `tailwindcss` import - No improvement

### Root Cause Analysis

The issue appears to be deeper than just color format. Even with:

- Correct HSL format in CSS variables
- Proper `@theme inline` mapping
- Inline styles with explicit colors

The computed colors are still incorrect, suggesting:

1. Something is overriding the styles with lower specificity colors
2. Tailwind v4's CSS processing might have a bug or configuration issue
3. There may be a conflict between multiple theme definitions or PostCSS processing
