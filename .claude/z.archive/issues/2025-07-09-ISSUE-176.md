## Summary

The E2E test setup script (`apps/e2e/scripts/test-setup.sh`) is incorrectly managing the database state. When running `pnpm test:e2e`, the script starts/stops Supabase which results in an empty database, causing tests to fail because critical tables (user_account_workspace, user_accounts, testimonials, accounts) are missing. The E2E tests share the same database as the main application, so the script should not be resetting the database state.

- **Application Version**: 2.11.0
- **Environment**: development

- **Node Version**: v22.16.0
- **Database**: PostgreSQL 15.8.1.044 (Supabase)
- **Last Working**: Unknown

## Reproduction Steps

1. Ensure Supabase is running with a populated database
2. Run `pnpm test:e2e`
3. The test setup script will restart Supabase
4. Tests fail with missing database tables error

## Expected Behavior

The E2E test setup script should:

- Check if Supabase is already running
- Use the existing database state without restarting
- Only verify that required services are accessible
- Not modify or reset the database

## Actual Behavior

The test setup script:

- Stops and restarts Supabase when detecting port conflicts
- This causes the database to be empty after restart
- Tests fail because critical tables are missing
- Error: "❌ Missing critical database objects: user_account_workspace user_accounts testimonials accounts"

## Initial Analysis

The issue occurs because:

1. The test setup script detects Supabase is not running (line 56) or email service is not accessible (line 68)
2. It then starts/restarts Supabase with `npx supabase start`
3. This appears to start with an empty database instead of using the existing database with migrations
4. The script expects certain tables to exist but finds none

The script comment on line 73-74 mentions "Skip migration check since we're using shared development database" but the behavior doesn't match this expectation.

## Suggested Investigation Areas

1. The script should not restart Supabase if it's already running properly
2. When Supabase needs to be started, it should preserve existing database state
3. Consider removing the automatic restart logic and requiring manual database setup
4. Investigate why `npx supabase start` results in an empty database

## Additional Context

- The correct way to reset the database is using `/workflow local-db-reset` or `pnpm supabase:web:reset`
- E2E tests are designed to use the same database as the main application
- The test setup script should be minimal and not modify database state
