# Task: Refactor /test command for deterministic execution with less LLM reliance

## Problem Statement

The current `/test` command implementation relies heavily on LLM agents (test-orchestrator → unit-test-agent → e2e-parallel-agent) which introduces unpredictability and flakiness. Complex bash conditionals require approval prompts, and test output parsing using regex is unreliable.

## Current Issues

1. **Over-reliance on LLM agents** - 3 layers of agents with complex prompts and decision-making
2. **Complex bash conditionals** - Many approval-required commands that could fail
3. **Ambiguous parsing** - Agents parsing test output with regex instead of structured data
4. **Unclear delegation flow** - Task tool delegation adds unpredictability
5. **Slow execution** - Multiple LLM round-trips during test execution

## Proposed Solution

### 1. Replace LLM Agents with Shell Scripts

Create deterministic bash/Node.js scripts:
- `test-runner.sh` - Main orchestrator script
- `run-unit-tests.sh` - Unit test executor with structured output
- `run-e2e-parallel.sh` - E2E shard coordinator
- `parse-test-results.js` - Node.js script for reliable JSON parsing

### 2. Use Structured Output Formats

- Configure test runners for JSON output:
  - Vitest: `--reporter=json`
  - Playwright: `--reporter=json`
- Parse with proper JSON parsers, not regex

### 3. Implement State Machine Approach

```
START → CHECK_INFRA → RUN_UNIT → RUN_E2E → AGGREGATE → REPORT
         ↓ fail        ↓ fail      ↓ fail
         FIX_INFRA     REPORT      REPORT
```

### 4. Simplify Command Interface

```bash
# /test command just calls:
.claude/scripts/test-runner.sh --mode=all --debug=$DEBUG_TEST
```

### 5. Use Process Managers for Parallelization

- Use `concurrently` or `npm-run-all` for parallel shards
- Or Node.js child_process with proper process management
- Avoid complex bash background job management

### 6. Implement Proper Status Tracking

JSON status file tracking:
```json
{
  "status": "running",
  "phase": "unit_tests",
  "unit": { "total": 0, "passed": 0, "failed": 0 },
  "e2e": { "shards": {}, "total": 0, "passed": 0 },
  "startTime": "2024-01-15T10:00:00Z"
}
```

### 7. Create Deterministic Error Handlers

```javascript
const ERROR_FIXES = {
  'ECONNREFUSED.*55321': 'cd apps/e2e && npx supabase start',
  'address already in use.*3000': 'pkill -f next-server',
  'webServer.*timeout': 'npx supabase status && npx supabase start'
}
```

### 8. Proposed Architecture

```
/test command
    ↓
test-wrapper.sh (simple bash)
    ↓
test-controller.js (Node.js orchestrator)
    ├── check-infrastructure()
    ├── runUnitTests() → vitest --reporter=json
    ├── runE2EShards() → parallel playwright calls
    └── generateReport() → structured JSON output
         ↓
Claude interprets results (optional)
```

### 9. Use NPM Scripts as Building Blocks

```json
{
  "test:all": "run-s test:infra test:unit test:e2e:parallel test:report",
  "test:infra": "node .claude/scripts/check-infra.js",
  "test:e2e:parallel": "node .claude/scripts/run-shards.js"
}
```

### 10. Limit LLM Involvement

If LLM is still needed, limit to:
- Interpreting final results
- Suggesting fixes for failures
- Answering questions about test output
- NOT for orchestrating execution

## Benefits

1. **Predictable execution** - Same inputs = same outputs
2. **Faster execution** - No LLM round-trips during test runs
3. **Better debugging** - Clear logs and state files
4. **Easier maintenance** - Simple scripts vs complex agent prompts
5. **Progressive enhancement** - LLM can still help interpret results
6. **No approval prompts** - Direct script execution
