## Overview
Implement two new GitHub Actions workflows to automate the validation and promotion process from dev to staging, eliminating the current manual gap after dev deployment completes.

## Problem
Currently, after the `Deploy to Dev` workflow completes successfully, nothing happens automatically. The promotion from dev → staging requires manual GitHub CLI commands, and there's no automated validation to ensure dev is ready for promotion.

## Solution
Add two new workflows that will automatically validate the dev deployment and create a promotion PR when ready.

### 1. Dev Integration Tests Workflow
**File**: `.github/workflows/dev-integration-tests.yml`
**Trigger**: Automatically after "Deploy to Dev" workflow completes
**Purpose**: Comprehensive validation of the deployed dev environment

**Components to implement**:
- [ ] API contract testing - Validate all API endpoints are responding
- [ ] Integration test suite - Run Playwright tests tagged with `@integration`
- [ ] Security quick scan - Check security headers and sensitive paths
- [ ] Performance baseline capture - Record Lighthouse metrics
- [ ] Test result aggregation and summary generation

**Duration**: ~15-20 minutes

### 2. Dev Promotion Readiness Workflow
**File**: `.github/workflows/dev-promotion-readiness.yml`
**Trigger**: After successful integration tests
**Purpose**: Assess readiness and automate PR creation

**Components to implement**:
- [ ] Quality gate assessment:
  - Success rate ≥ 80%
  - Code coverage ≥ 80%
  - No recent failures (last 6h)
  - Change size < 100 files
- [ ] Automatic draft PR creation to staging branch
- [ ] Changelog generation from commits
- [ ] Promotion scorecard in PR description
- [ ] Stale deployment detection (> 24h warning)

## Implementation Steps

### Phase 1: Core Implementation
1. [ ] Review and test the created workflow files
2. [ ] Add `@integration` tags to existing Playwright tests
3. [ ] Configure required GitHub secrets:
   - `E2E_TEST_EMAIL`
   - `E2E_TEST_PASSWORD`
   - Ensure `GITHUB_TOKEN` has workflow permissions
4. [ ] Test workflow triggers with a dev deployment

### Phase 2: Testing & Refinement
1. [ ] Push test commit to dev branch
2. [ ] Verify integration tests trigger and run correctly
3. [ ] Confirm promotion readiness assessment works
4. [ ] Validate draft PR is created with correct content
5. [ ] Adjust quality gate thresholds based on results

## Success Criteria
- [ ] Dev deployments automatically trigger integration tests
- [ ] Successful tests trigger promotion readiness assessment
- [ ] Draft PRs are created automatically when quality gates pass
- [ ] Manual approval is still required (GitOps maintained)
- [ ] Total automation time < 20 minutes
- [ ] Clear visibility into why promotion is/isn't ready

## Benefits
- **Time Saved**: ~30 minutes per deployment cycle
- **Quality**: Issues caught before staging
- **Visibility**: Clear metrics and scorecards
- **Audit Trail**: GitOps via PRs maintained
- **Reduced Toil**: No manual PR creation needed

## Flow Diagram
```
Push to dev → Deploy to Dev → Integration Tests → Promotion Assessment → Draft PR → Manual Review → Merge → Deploy to Staging
```
