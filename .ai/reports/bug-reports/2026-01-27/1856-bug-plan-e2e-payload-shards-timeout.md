# Bug Fix: E2E Payload Shards (7, 8, 9) Timeout Due to Missing Seeded Admin User

**Severity**: high

## Quick Reference

- **Root Cause**: Payload admin user (`michael@slideheroes.com`) is not seeded after migrations run in e2e-sharded workflow, causing authentication failures during global-setup
- **Fix Approach**: Add Payload admin user seeding step to `.github/workflows/e2e-sharded.yml` after migrations complete
- **Estimated Effort**: small
- **Breaking Changes**: no

## Solution Design

### Problem Recap

E2E test shards 7, 8, 9 (Payload CMS tests) timeout at 15 minutes because the Payload admin user is not created after database migrations run in the CI workflow. When `global-setup.ts` attempts to authenticate via `loginToPayloadWithRetry()`, Payload returns no token since the user doesn't exist in the empty `payload.users` table.

**Evidence from diagnosis**:
- Migrations run successfully (✅ `Run Payload CMS migrations` step exists)
- Payload login attempts fail (❌ "no token received" after 3 retries)
- Global setup skips Payload authentication for payload-admin user
- Tests timeout at 15 minutes (default Playwright timeout)

For full details, see diagnosis issue #1855.

### Solution Approaches Considered

#### Option 1: Add Payload User Seed Step to CI Workflow ⭐ RECOMMENDED

**Description**: Add a new step to `.github/workflows/e2e-sharded.yml` that creates the Payload admin user immediately after migrations complete. This uses direct SQL insert to create the user with proper bcrypt password hashing.

**Pros**:
- Simple one-time operation (5-10 seconds)
- No new dependencies required
- Uses existing Supabase client already available in workflow
- Idempotent (uses `WHERE NOT EXISTS` to prevent duplicates)
- Minimal workflow changes
- Follows existing seeding patterns in the codebase

**Cons**:
- Direct SQL requires knowing Payload's schema
- Password hashing must match Payload's requirements

**Risk Assessment**: low - Direct SQL insert is straightforward; test validation catches issues

**Complexity**: simple - Single SQL statement with basic error handling

#### Option 2: Use Payload API to Create Admin User

**Description**: Create admin user via Payload's REST API endpoint after migrations (requires waiting for Payload to start).

**Pros**:
- More robust (uses official API)
- Schema-agnostic
- Would work if Payload schema changes

**Cons**:
- Requires Payload server to be running and healthy
- More complex orchestration (health check, retry loop)
- Slower (server startup overhead)
- May require special flags/admin tokens

**Why Not Chosen**: Option 1 is simpler and Payload isn't running during migrations, so API approach would require starting the Payload server twice (once for migrations, once for API calls)

#### Option 3: Pre-seed Admin User in Payload Migration

**Description**: Include Payload admin user creation as part of the migration files instead of a separate workflow step.

**Pros**:
- Migrations are version-controlled
- Would work automatically in all environments

**Cons**:
- Payload migrations generated by Payload CLI (manual editing risky)
- Hard to maintain if Payload schema changes
- Duplicates what Payload's seed system should handle

**Why Not Chosen**: Payload migrations should be generated, not manually edited; separate workflow step is cleaner

### Selected Solution: Add Payload User Seed Step to CI Workflow

**Justification**: This is the simplest, safest approach that directly addresses the root cause. The admin user seed is an E2E test infrastructure concern (not a schema concern), so it belongs in the workflow, not migrations. The SQL is straightforward, idempotent, and follows patterns already used in the codebase for seeding operations.

**Technical Approach**:

1. **Create Payload admin user** using direct SQL insert with bcrypt password hashing
2. **Execute after migrations** but before Payload server starts
3. **Make idempotent** to safely re-run if needed
4. **Minimal changes** - Single step added to `.github/workflows/e2e-sharded.yml`

**Architecture Changes**: None - leveraging existing infrastructure and patterns

**Migration Strategy**: Not needed - this is a test data seeding operation
