### Technical Implementation

```yaml
- name: Seed Payload admin user
  run: |
    echo "🌱 Seeding Payload admin user..."
    if pnpm --filter payload seed:run -c users; then
      echo "✅ Payload admin user created"
    else
      echo "ℹ️  Payload admin user may already exist"
      # Verify user exists in database
      psql "$DATABASE_URI" -c "SELECT email FROM payload.users WHERE email = 'michael@slideheroes.com'" | grep -q michael@slideheroes.com && \
        echo "✅ Payload admin user verified in database" || \
        (echo "❌ Payload admin user not found" && exit 1)
    fi
  env:
    DATABASE_URI: postgresql://postgres:postgres@localhost:54522/postgres
    PAYLOAD_SECRET: test_payload_secret_for_e2e_testing
    SEED_USER_PASSWORD: ${{ secrets.E2E_ADMIN_PASSWORD }}
```

## Root Cause

The Payload migrations step creates empty database tables but doesn't seed any data. Global-setup's `loginToPayloadWithRetry()` fails because:
1. No user exists in `payload.users` table
2. Login API returns 401/403 (invalid credentials)
3. After 3 retries with exponential backoff, throws error
4. Workflow times out at 15 minutes

## Why This Fix Works

1. **Payload Local API** - Uses `getPayload()` to initialize Payload directly without a running HTTP server
2. **Proper Password Hashing** - Payload's internal hashing mechanism (salt+hash) is used automatically
3. **Seed Engine** - Leverages existing infrastructure (`seed:run -c users`) for user creation
4. **Idempotent** - Handles cases where user already exists gracefully
