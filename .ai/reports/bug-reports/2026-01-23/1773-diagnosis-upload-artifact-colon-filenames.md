## Summary

The Deploy to Staging workflow fails at the "Upload artifacts" step because Next.js/Turbopack produces build output files with colons (`:`) in the filename (e.g., `[externals]_node:child_process_261e8bca._.js`). GitHub Actions `upload-artifact@v6` permanently rejects files containing colons due to NTFS file system compatibility restrictions.

This is a **new issue** distinct from Issues #1763 (duplicate --cache-dir) and #1769 (hidden files excluded), both of which have been fixed.

## Root Cause Analysis

### Identified Root Cause

**Summary**: GitHub Actions `upload-artifact@v6` rejects files with colons (`:`) in filenames, and Next.js/Turbopack generates server chunk files with `node:` protocol references in their names.

**Detailed Explanation**:
1. Next.js server-side rendering uses Node.js built-in modules like `child_process`
2. When Turbopack bundles these, it creates chunk files with names that include `node:` (e.g., `[externals]_node:child_process_261e8bca._.js`)
3. GitHub Actions upload-artifact has a permanent restriction on characters that are invalid on NTFS filesystems, including colons
4. This restriction is by design and will not change in any version of upload-artifact

**Supporting Evidence**:
- Error message explicitly states colon is not allowed due to NTFS limitations
- File path shows the pattern: `[externals]_node:child_process_261e8bca._.js`
- This is documented GitHub Actions behavior, not a bug

### How This Causes the Observed Behavior

1. Build completes successfully, generating `.next/server/chunks/` with colon-containing files
2. Upload artifacts step begins processing 2050 files
3. When it encounters a file with `:` in the name, it fails immediately
4. The entire artifact upload fails, skipping subsequent deployment jobs

### Confidence Level

**Confidence**: High

**Reasoning**: The error message is explicit about the cause (colon in filename), the file exists and has that name, and this is documented GitHub Actions behavior. There is no ambiguity.

## Fix Approach (High-Level)

**Recommended Solution: Tar archive approach**

Wrap the build artifacts in a tar archive before uploading. This preserves all filenames (including colons) inside the archive while the archive itself has a safe name.

```yaml
- name: Archive build artifacts
  run: |
    tar -cvf build-artifacts.tar \
      apps/web/.next \
      apps/web/public \
      apps/payload/dist \
      packages/*/dist \
      build-manifest.json

- name: Upload artifacts
  uses: actions/upload-artifact@v6
  with:
    name: build-artifacts-${{ inputs.environment }}-${{ inputs.commit_sha }}
    path: build-artifacts.tar
    retention-days: 7
```

The download step would need corresponding extraction:
```yaml
- name: Download artifacts
  uses: actions/download-artifact@v7
  with:
    name: build-artifacts-...
    path: .

- name: Extract artifacts
  run: tar -xvf build-artifacts.tar
```

**Alternative Solutions Considered**:
1. **Filename sanitization** - Rename files with colons before upload, rename back after download. More fragile and could break references.
2. **Exclude problematic files** - Not viable, these are required server chunks.
3. **Use actions/cache instead** - Not suitable for cross-job artifact sharing.

## Diagnosis Determination

**Root cause confirmed**: GitHub Actions upload-artifact@v6 permanently rejects filenames containing colons (`:`) for NTFS compatibility, and Next.js/Turbopack generates server chunk files with `node:` in their names.

**Fix path clear**: Use tar archiving to encapsulate the build artifacts, preserving internal filenames while presenting a safe external filename to GitHub Actions.

## Additional Context

- This fix should be applied to all workflows that upload `.next` directories: `artifact-sharing.yml`, `reusable-build.yml`, `e2e-sharded.yml`.
