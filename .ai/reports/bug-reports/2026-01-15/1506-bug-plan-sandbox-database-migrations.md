# Bug Fix: Remote Supabase sandbox database tables not created during Alpha orchestrator execution

**Related Diagnosis**: #1503
**Severity**: high
**Bug Type**: integration
**Risk Level**: medium
**Complexity**: moderate

## Quick Reference

- **Root Cause**: Three interrelated issues: (1) timing - `resetSandboxDatabase()` runs at startup BEFORE features create migrations, (2) missing authentication - `SUPABASE_ACCESS_TOKEN` not configured in `.env`, (3) invalid credentials or project configuration
- **Fix Approach**: Defer database reset to post-feature sync phase, add SUPABASE_ACCESS_TOKEN to environment setup, implement retry with auth validation
- **Estimated Effort**: medium
- **Breaking Changes**: no

## Solution Design

### Problem Recap

The Alpha Autonomous Coding workflow completes spec implementation (Spec #1362), but the remote Supabase sandbox database contains no tables. Database tasks create migration files but never apply them because:

1. `resetSandboxDatabase()` in `orchestrator.ts` runs at startup, BEFORE features implement and create migrations
2. New migrations generated by features are never pushed to the remote database
3. `SUPABASE_ACCESS_TOKEN` is not configured in `.env`, preventing Supabase CLI from authenticating
4. Database connection may fail due to invalid credentials

This blocks human-in-the-loop review via E2B preview URLs and database-dependent features.

For full details, see diagnosis issue #1503.

### Solution Approaches Considered

#### Option 1: Move Database Reset to Post-Feature Sync ⭐ RECOMMENDED

**Description**:
- Keep `resetSandboxDatabase()` at startup for schema cleanup
- Add new `syncFeatureMigrations()` function after each feature completes
- Push accumulated migrations from features to remote database
- Seed database after all migrations are applied

**Pros**:
- Captures migrations created by features (core issue)
- Maintains separation of concerns (reset vs sync)
- Allows incremental migration application per feature
- Minimal changes to existing orchestrator flow
- Backwards compatible with existing database utilities

**Cons**:
- Adds complexity to orchestrator work loop
- Multiple migration sync operations (one per feature)
- Requires careful error handling for partial migrations

**Risk Assessment**: low - approach is additive, doesn't break existing reset logic

**Complexity**: moderate - need to add sync mechanism and integrate into feature loop

#### Option 2: Run Full Database Reset After All Features

**Description**:
- Skip database operations at startup
- Run complete reset (schema + migrations + seeding) after all features complete
- Simpler but defers all DB work to end

**Pros**:
- Single database operation (simpler)
- Guaranteed to capture all migrations
- Clear separation (features first, DB second)

**Cons**:
- Long database wait at end (blocks review)
- Can't test DB features during implementation
- Features can't validate against real data
- Doesn't support incremental feature testing

**Why Not Chosen**: Blocks human-in-the-loop review. Preview URL won't work until all features are done.

#### Option 3: Branch Database State Per Feature

**Description**:
- Create database snapshot per feature
- Reset/branch database for each feature's work
- Merge back after feature completes

**Pros**:
- True feature isolation
- Each feature can test independently

**Cons**:
- Complex branching logic (PostgreSQL doesn't have built-in branching)
- Significant performance overhead
- Over-engineered for current needs

**Why Not Chosen**: Too complex for immediate bug fix. Consider for future enhancement.

### Selected Solution: Move Database Reset to Post-Feature Sync

**Justification**:
This approach directly addresses the root cause (migrations created during feature work are never applied). It maintains the existing database reset pattern for initial cleanup while adding a new sync mechanism for feature-generated migrations. This enables:
- Features to create database changes (migrations, schemas)
- Automatic migration application after each feature
- Human review of working code with real database state
- Incremental validation as features complete

**Technical Approach**:
1. Keep existing `resetSandboxDatabase()` at startup (cleans public schema)
2. Add `SUPABASE_ACCESS_TOKEN` to environment setup in `.env.example` and initialization
3. After each feature completes, call new `syncFeatureMigrations(sandbox)` function
4. New function pushes migrations from feature branch to remote database
5. Seed database after final feature if needed

**Architecture Changes**:
- Add `syncFeatureMigrations()` to `database.ts`
- Call from orchestrator's main work loop after feature completes
- Add token validation to sandbox setup in `sandbox.ts`
- Document Supabase configuration requirements

**Migration Strategy**:
- Existing database state will be cleared on first reset
- Features can assume clean schema after reset
- New migrations will accumulate and be applied incrementally

## Risk Assessment

**Overall Risk Level**: medium

**Potential Risks**:

1. **Migration Sync Fails (Auth Error)**: Existing auth issue persists
   - **Likelihood**: medium
   - **Impact**: medium (blocks database access, but doesn't break orchestrator)
   - **Mitigation**: Implement graceful error handling - allow orchestrator to continue, log warnings clearly. Add token validation at startup so users know immediately if credentials are wrong.

2. **Timing Issues**: Migrations not fully applied before next feature starts
   - **Likelihood**: low
   - **Impact**: high (inconsistent database state)
   - **Mitigation**: Add explicit wait for migration completion. Check database status before returning from sync function. Add health check between features.

3. **Multiple Sandboxes Conflict**: If multiple orchestrators run simultaneously, migrations could conflict
   - **Likelihood**: low (single orchestrator runs at a time)
   - **Impact**: medium (database corruption)
   - **Mitigation**: Keep existing lock mechanism. Each orchestrator gets exclusive database access during reset. Document that only one orchestrator should run at a time.

4. **Partial Migrations Applied**: Migration pushed but not fully applied
   - **Likelihood**: low
   - **Impact**: medium (incorrect schema state)
   - **Mitigation**: Verify migration completion by querying database schema after sync. If verification fails, log error and alert user.

5. **Token Leakage**: `SUPABASE_ACCESS_TOKEN` hardcoded in environment
   - **Likelihood**: low (should use secrets management)
   - **Impact**: high (security)
   - **Mitigation**: Token only used in CI/orchestrator environment. Document that production should use environment secrets. Never commit token to git.

**Rollback Plan**:

If this fix causes database issues in production:

1. Stop orchestrator: `pkill -f spec-orchestrator`
2. Disable migration sync: Comment out `syncFeatureMigrations()` call in orchestrator.ts
3. Revert to previous version: `git checkout HEAD~1 .ai/alpha/scripts/lib/orchestrator.ts`
4. Rebuild orchestrator: `tsx .ai/alpha/scripts/spec-orchestrator.ts`
5. Clear sandbox database: `psql $SUPABASE_SANDBOX_DB_URL -c "DROP SCHEMA IF EXISTS public CASCADE;"`
6. Run orchestrator again (will use old behavior)

**Monitoring** (if deployed to CI):
- Monitor orchestrator logs for "Migration sync failed" messages
- Alert if more than 2 consecutive feature failures
- Track database size growth (should be gradual, not sudden jumps)
