## Root Cause Analysis

### Identified Root Cause

**Summary**: ES Module import hoisting causes environment variables to be captured at module load time, BEFORE the custom `loadEnvFile()` function runs.

**Detailed Explanation**:

In ES Modules, **all import statements are hoisted and executed before any top-level code**. The code in `spec-orchestrator.ts` appears to call `loadEnvFile()` before the imports:

```typescript
// spec-orchestrator.ts (textual order)
function loadEnvFile(): void { ... }
loadEnvFile();  // Line 73 - appears BEFORE imports

import { orchestrate } from "./lib/index.js";  // Line 76
```

However, ESM execution order is:

1. **FIRST**: All `import` statements execute (hoisted)
   - `import { orchestrate } from "./lib/index.js"` runs
   - This imports `./lib/environment.js` which evaluates:
     ```typescript
     export const SUPABASE_ACCESS_TOKEN = process.env.SUPABASE_ACCESS_TOKEN;  // Line 97
     export const SUPABASE_SANDBOX_PROJECT_REF = process.env.SUPABASE_SANDBOX_PROJECT_REF;  // Lines 98-99
     ```
   - At this moment, `process.env.SUPABASE_ACCESS_TOKEN` is `undefined` (not yet loaded from .env)

2. **THEN**: Top-level code runs
   - `loadEnvFile()` executes and sets `process.env.SUPABASE_ACCESS_TOKEN`
   - But the `const` exports in `environment.ts` already captured `undefined`

**Supporting Evidence**:
- Stack trace shows error at `sandbox.ts:107` inside `createSandbox()`
- The warning "Supabase CLI not configured" appears early, confirming the race condition
- Both tokens ARE present in `.env` file, yet the validation fails
- The `validateSupabaseTokensRequired()` function reads from the already-captured `SUPABASE_ACCESS_TOKEN` const, not from `process.env` directly

### How This Causes the Observed Behavior

1. User runs `tsx spec-orchestrator.ts 1362`
2. ESM hoists imports - `environment.ts` loads first
3. `environment.ts` captures `SUPABASE_ACCESS_TOKEN = undefined` (env not loaded yet)
4. `loadEnvFile()` runs and sets `process.env.SUPABASE_ACCESS_TOKEN` correctly
5. `checkEnvironment()` at line 803 reads the stale `SUPABASE_ACCESS_TOKEN` const → warns "not configured"
6. `createSandbox()` calls `validateSupabaseTokensRequired()` which reads the same stale const → throws error
7. Error message misleadingly suggests tokens are missing when they're actually in .env

### Confidence Level

**Confidence**: High

**Reasoning**:
- The stack trace shows the exact failure point in validation code
- The .env file provably contains the tokens
- ES Module hoisting behavior is well-documented and deterministic
- The same pattern (const exports capturing process.env at load time) is used throughout environment.ts

## Fix Approach (High-Level)

**Option A (Recommended)**: Convert the const exports to getter functions that read `process.env` lazily at call time:

```typescript
// Before (captures at load time - broken):
export const SUPABASE_ACCESS_TOKEN = process.env.SUPABASE_ACCESS_TOKEN;

// After (reads at call time - works):
export function getSupabaseAccessToken(): string | undefined {
  return process.env.SUPABASE_ACCESS_TOKEN;
}
```

**Option B**: Use dynamic imports to defer loading environment.ts until after loadEnvFile() completes:

```typescript
loadEnvFile();
const { orchestrate } = await import("./lib/index.js");
```

## Diagnosis Determination

The root cause is definitively identified: **ES Module import hoisting** causes `environment.ts` to capture environment variables before the custom `.env` loader runs. The fix requires changing from eager const exports to lazy getter functions for any environment variables that depend on runtime loading.

## Additional Context

This is a common ESM gotcha that doesn't occur in CommonJS (where `require()` can be placed anywhere). The code was likely written assuming CJS-like synchronous loading behavior.

The fail-fast validation recently added in #1511 surfaced this pre-existing race condition by making the failure immediate and clear rather than silently proceeding with missing tokens.
