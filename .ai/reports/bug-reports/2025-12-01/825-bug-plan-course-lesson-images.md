**Severity**: medium

## Quick Reference

- **Root Cause**: Code doesn't use the `lesson.thumbnail` object from Payload API; always returns placeholder instead
- **Fix Approach**: Update `CourseDashboardClient.tsx` to use `lesson.thumbnail?.url` with fallback to placeholder
- **Estimated Effort**: small
- **Breaking Changes**: no

## Solution Design

### Problem Recap

The course dashboard at `/home/course` displays lesson cards with empty image areas. After Payload CMS schema evolved from `featured_image_id` to `thumbnail` (now an expanded media object), the frontend code was never updated. Code contains an explicit comment: "Since featured_image_id doesn't exist, use placeholder" even though the API now returns `lesson.thumbnail` with valid URLs.

Secondary issue: The placeholder SVG file (`default-lesson.svg`) is empty (0 bytes), so even the placeholder fails to render.

For full details, see diagnosis issue #824.

### Solution Approaches Considered

#### Option 1: Use Thumbnail URL with Placeholder Fallback ⭐ RECOMMENDED

**Description**: Update the image rendering logic to check for `lesson.thumbnail?.url` first, then fall back to the placeholder image.

**Pros**:
- Minimal code change (2-3 lines)
- Uses the actual thumbnail data already returned by API
- Clear intent and easy to maintain
- Leverages existing placeholder logic as fallback
- No breaking changes to component interface

**Cons**:
- Requires understanding Payload media object structure
- Minimal risk

**Risk Assessment**: Low - purely additive change to use existing data

**Description**: Create a new valid SVG for the placeholder instead of using a 0-byte file.

**Pros**:

- Improves user experience when thumbnail isn't available

**Cons**:

### Selected Solution: Use Thumbnail URL with Placeholder Fallback

**Justification**: This is the correct fix because:
1. The Payload API already returns `lesson.thumbnail` data - we should use it
2. Code currently ignores this valid data despite explicit field in CMS
3. Minimal, surgical change that fixes the root cause
4. Maintains backward compatibility with placeholder fallback
5. Aligns with schema migration intent

**Technical Approach**:
- Read `lesson.thumbnail` from the Payload API response
- Extract URL from the thumbnail object if it exists
- Apply existing URL transformation logic (`_transformImageUrl`)
- Fall back to placeholder only if thumbnail is missing/null

**Architecture Changes**: None - uses existing component structure

### Payload CMS Schema Reference

The `thumbnail` field in CourseLessons is defined as:

```typescript
{
  name: "thumbnail",
  type: "upload",
  relationTo: "media",
  admin: {
    description: "Thumbnail image for this lesson",
  },
}
```

When expanded in the API response, it returns: `{ url: string, ...otherMediaFields }`

### URL Transformation Function

The `_transformImageUrl()` function at lines 23-38 already handles:
- Cloudflare R2 URLs → custom domain transformation
- Bare filenames → adding custom domain prefix
- Full URLs → pass through as-is

This function should be applied to thumbnail URLs to ensure consistent transformation.
