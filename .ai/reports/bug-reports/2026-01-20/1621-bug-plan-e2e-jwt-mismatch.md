# Bug Fix: E2E Sharded Workflow JWT Secret Mismatch

**Related Diagnosis**: #1615
**Severity**: high
**Bug Type**: integration
**Risk Level**: low
**Complexity**: simple

## Quick Reference

- **Root Cause**: Workflow uses hardcoded HS256 JWT keys while fresh Supabase instances generate ES256-based keys, causing `AuthApiError: invalid JWT: signing method HS256 is invalid`
- **Fix Approach**: Extract actual JWT keys from Supabase after startup using `supabase status -o env` and set them in workflow environment
- **Estimated Effort**: small
- **Breaking Changes**: no

## Solution Design

### Problem Recap

The e2e-sharded workflow (`.github/workflows/e2e-sharded.yml`) sets hardcoded JWT keys on lines 127-129:

```yaml
SUPABASE_ANON_KEY: 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...'
SUPABASE_SERVICE_ROLE_KEY: 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...'
```

These keys use the HS256 algorithm from the "supabase-demo" template. However, when each shard runs `supabase start`, it generates a fresh Supabase instance with ES256-based keys. The mismatch causes authentication to fail:

```
AuthApiError: invalid JWT: signing method HS256 is invalid
```

This breaks shards 2-11 which require authentication for test user creation in globalSetup.

For full details, see diagnosis issue #1615.

### Solution Approaches Considered

#### Option 1: Extract Keys from Supabase After Startup ⭐ RECOMMENDED

**Description**: After `supabase start`, run `supabase status -o env` to extract the actual keys generated by the fresh instance, then export them for use by E2E tests.

**Pros**:
- Fixes root cause directly - uses the actual keys that Supabase generates
- No manual key management or hardcoding needed
- Works for both HS256 and ES256 algorithms (future-proof)
- Simple, elegant, requires only 4 lines of shell script
- Zero performance impact
- Aligns with Supabase best practices

**Cons**:
- Requires both setup-server and e2e-shards jobs to extract keys (slight duplication)
- Must run AFTER `supabase start` but BEFORE tests run

**Risk Assessment**: low - this is standard Supabase workflow practice. Supabase CLI guarantees these variables are always available after startup.

**Complexity**: simple - just shell commands to extract and export variables

#### Option 2: Use Supabase Admin Client API

**Description**: After startup, query the Supabase Admin API to fetch generated keys programmatically.

**Why Not Chosen**: The `supabase status -o env` command already provides exactly what we need with zero dependencies. Adding API calls would be over-engineering.

#### Option 3: Mock/Stub JWT Validation

**Description**: Disable JWT validation in test environment or mock Supabase Auth.

**Why Not Chosen**: We need real JWT validation to test auth workflows properly. Mocking defeats the purpose of E2E tests for authentication flows.

### Selected Solution: Extract Keys from Supabase Status

**Justification**: This approach directly addresses the root cause by using the actual keys that Supabase generates instead of hardcoded ones. It's the standard practice recommended by Supabase for local development and CI/CD pipelines.

**Technical Approach**:
1. After `supabase start` completes, run `supabase status -o env` to output all environment variables
2. Parse the output to extract `ANON_KEY` and `SERVICE_ROLE_KEY`
3. Export them as `SUPABASE_ANON_KEY`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, and `SUPABASE_SERVICE_ROLE_KEY`
4. These variables are then available for all subsequent steps

**Architecture Changes**: None - this is purely operational, no code changes needed.

## Notes

**Key insight from diagnosis**: The real issue is that Supabase generates ES256-based JWT keys by default in local instances, while the workflow provides HS256 keys. Rather than trying to force HS256 (which breaks Supabase's security model), we extract the actual keys that Supabase generates.

**Why this wasn't caught before**: The hardcoded HS256 keys worked with older Supabase versions that used HS256 by default. Recent Supabase versions switched to ES256 for better security, but the workflow wasn't updated.
