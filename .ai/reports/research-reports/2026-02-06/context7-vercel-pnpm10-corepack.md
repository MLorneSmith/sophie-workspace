# Context7 Research: Vercel pnpm 10.x Corepack Configuration

**Date**: 2026-02-06
**Agent**: context7-expert
**Libraries Researched**: vercel/vercel, websites/vercel, websites/pnpm_io, llmstxt/vercel_llms_txt

## Query Summary

Investigated how Vercel handles pnpm 10.x with corepack, the `ENABLE_EXPERIMENTAL_COREPACK` environment variable, and the root cause of the `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH` error related to `patchedDependencies`.

## Findings

### 1. Root Cause: Vercel's pnpm Version Detection Race Condition

The deploy log reveals the exact problem:

```
Detected `pnpm-lock.yaml` 9 which may be generated by pnpm@9.x or pnpm@10.x
Using pnpm@9.x based on project creation date
To use pnpm@10.x, manually opt in using corepack
Running "install" command: `corepack enable && corepack prepare pnpm@10.14.0 --activate && pnpm install --frozen-lockfile`...
```

**What happens**: Vercel detects the lockfile format version 9.0 (used by both pnpm 9.x and 10.x). It then **defaults to pnpm 9.x** based on project creation date. However, the custom `installCommand` runs corepack to activate pnpm 10.14.0 — but by this time, Vercel has **already initialized the pnpm environment with pnpm 9.x settings**. The `patchedDependencies` field in `package.json` uses pnpm 10.x format, which doesn't match what pnpm 9.x wrote into its internal config state, causing `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH`.

### 2. The `ENABLE_EXPERIMENTAL_COREPACK` Environment Variable

From Vercel docs (vercel.com/docs/builds/configure-a-build):

> **Corepack allows projects to pin a specific package manager version.** This is achieved by setting an environment variable and then defining the desired package manager and version in the package.json file.

The critical piece: **Set `ENABLE_EXPERIMENTAL_COREPACK=1` as a Vercel environment variable.** This tells Vercel to use corepack **from the very start of the build process**, before any install command runs. This means Vercel will:

1. Read the `"packageManager": "pnpm@10.14.0"` field from `package.json`
2. Use corepack to activate pnpm 10.14.0 **before** running any install command
3. Skip its own pnpm version detection heuristic

### 3. The Fix: Proper Configuration

**The `installCommand` with `corepack enable && corepack prepare` is NOT sufficient on its own.** The env var must be set because Vercel's build system initializes the package manager BEFORE executing the custom installCommand. The installCommand only controls what runs during the "install" phase, but Vercel's build orchestrator has already made pnpm version decisions by then.

#### Recommended Configuration:

**Step 1**: Set environment variable in Vercel Dashboard (or via CLI):
```
ENABLE_EXPERIMENTAL_COREPACK=1
```
Apply to: Production, Preview, and Development environments.

**Step 2**: Ensure `package.json` has the `packageManager` field:
```json
{
  "packageManager": "pnpm@10.14.0"
}
```

**Step 3**: Simplify `vercel.json` installCommand:
```json
{
  "installCommand": "pnpm install --frozen-lockfile"
}
```
Or remove `installCommand` entirely to let Vercel auto-detect.

### 4. Why `patchedDependencies` Specifically Triggers This

The `patchedDependencies` field in `package.json` is written into the lockfile settings section. When Vercel starts with pnpm 9.x but the lockfile was generated by pnpm 10.x, the internal representation of `patchedDependencies` differs between versions. The `--frozen-lockfile` flag enforces strict matching, so any config divergence causes `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH`.

The lockfile header confirms pnpm 10.x format:
```yaml
lockfileVersion: '9.0'
settings:
  autoInstallPeers: true
  excludeLinksFromLockfile: false
patchedDependencies:
  payload@3.75.0:
    hash: bc647ee89ca63bbf845c933d69096f88954779a83702d6a06275cdb4afe41ce8
    path: patches/payload@3.75.0.patch
```

### 5. Known Issues and Corepack Gotchas

From Vercel KB (vercel.com/kb/guide/corepack-errors-github-actions):
- Corepack versions bundled with Node.js may be outdated
- For Node.js 18+: upgrade to corepack@latest (`npm install -g corepack@latest`)
- The `ENABLE_EXPERIMENTAL_COREPACK` env var is the officially supported way to use corepack on Vercel

From pnpm.io CI documentation:
- The recommended CI pattern is: `corepack enable && corepack prepare pnpm@<version> --activate && pnpm install`
- In CI environments, `--frozen-lockfile` is enabled by default when a lockfile is present

## Key Takeaways

1. **Set `ENABLE_EXPERIMENTAL_COREPACK=1`** as a Vercel environment variable — this is the primary fix
2. The `installCommand` with `corepack enable && corepack prepare` runs **too late** — Vercel has already chosen a pnpm version
3. With `ENABLE_EXPERIMENTAL_COREPACK=1`, Vercel reads `packageManager` from `package.json` and uses that version from the start
4. The `installCommand` can then be simplified to just `pnpm install --frozen-lockfile` (or removed entirely)
5. The `patchedDependencies` mismatch is a symptom — the root cause is Vercel defaulting to pnpm 9.x
