## Findings

### E2B Template Creation Workflow

There are two approaches to creating E2B templates:

#### Approach 1: v1 Build System (Dockerfile-based) - Current Standard

1. **Initialize template**: `e2b template init` creates a basic `e2b.Dockerfile`
2. **Build template**: `e2b template build` builds from Dockerfile and generates `e2b.toml`
3. **Publish template**: `e2b template publish` makes it available to others

#### Approach 2: v2 Build System (SDK-based) - New Format

1. **Initialize v2 template**: `e2b template init-v2` creates a new Template SDK format
2. **Migrate existing**: `e2b template migrate` converts v1 to v2 format

### e2b.toml Configuration Format

The `e2b.toml` file is generated by the build command but can be manually created:

```toml
# This is a config for E2B sandbox template
template_id = "your-template-id"           # Generated after first build
dockerfile = "e2b.Dockerfile"              # Path to Dockerfile
template_name = "my-agent-sandbox"         # Lowercase, letters, numbers, dashes, underscores
start_cmd = "<your-start-command>"         # Optional: command on sandbox start
ready_cmd = "<your-ready-command>"         # Optional: must exit 0 for template ready
```

**Important**: The `template_id` is assigned by E2B during the first build. For new templates, omit it or leave empty.

### CLI Commands Reference

#### Authentication
```bash
e2b auth login
# Or use environment variable:
E2B_ACCESS_TOKEN=sk_e2b_... e2b template build
```

#### Initialize New Template
```bash
# v1 (Dockerfile-based)
e2b template init [-p, --path <path>]

# v2 (SDK-based) - newer format
e2b template init-v2 [options]
```

#### Build Template
```bash
e2b template build [options] [template]

Options:
  -p, --path <path>           # Root directory for command execution
  -d, --dockerfile <file>     # Dockerfile path (default: e2b.Dockerfile or Dockerfile)
  -n, --name <template-name>  # Template name (lowercase, alphanumeric, dashes, underscores)
  -c, --cmd <start-command>   # Command executed when sandbox starts
  --ready-cmd <ready-command> # Command that must exit 0 for template ready
  -t, --team <team-id>        # Team ID from E2B dashboard
  --config <e2b-toml>         # Path to e2b.toml config
  --cpu-count <cpu-count>     # CPUs for sandbox (default: 2)
  --memory-mb <memory-mb>     # Memory in MB (default: 512, must be even)
  --build-arg <args...>       # Build arguments (format: varname=value)
  --no-cache                  # Skip build cache
```

#### Publish Template
```bash
e2b template publish [options] [template]

Options:
  -p, --path <path>           # Root directory
  --config <e2b-toml>         # Config file path
  -s, --select                # Interactive template selection
  -t, --team <team-id>        # Team ID
  -y, --yes                   # Skip confirmation
```

#### List Templates
```bash
e2b template list [-t, --team <team-id>] [-f, --format <format>]
```

#### Delete Template
```bash
e2b template delete [options] [template]
```

### Dockerfile Format for E2B

E2B uses standard Dockerfiles with some conventions:

```dockerfile
# Option 1: Extend an E2B base image (recommended for code interpreter)
FROM e2bdev/code-interpreter:latest

# Install additional packages
RUN pip install cowsay

# Option 2: Start from any Debian-based image
FROM ubuntu:latest

# Install required packages
RUN apt-get update && apt-get install -y nodejs npm git

# Install global npm packages
RUN npm install -g pnpm @openai/codex
```

**Key Points:**
- Use Debian-based images (Ubuntu, Debian, etc.)
- E2B looks for `e2b.Dockerfile` or `Dockerfile` by default
- The `-d` flag can specify a custom Dockerfile path

## Key Takeaways

1. **For new templates**: Use `e2b template build -n <name>` without pre-existing e2b.toml
2. **The template_id is generated**: It is assigned by E2B on first successful build
3. **Do not pre-create e2b.toml**: Let the build command generate it
4. **Use Debian-based images**: Ubuntu is recommended
5. **Name constraints**: lowercase, letters, numbers, dashes, underscores only
6. **v2 system exists**: Use `init-v2` and `migrate` commands, but v1 still works
