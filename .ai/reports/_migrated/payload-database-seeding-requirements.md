**Date**: 2025-09-30  
**Author**: Database Analysis Assistant

**Priority**: High

## Executive Summary

This document provides comprehensive database requirements for implementing the Payload CMS seeding system in the SlideHeroes project. Based on analysis of ~316 records across 10+ collections with ~400 relationships, we recommend the **Payload Local API approach** for its automatic relationship handling, validation, and maintainability advantages over direct SQL insertion.

**Key Metrics**:
- **Collections**: 10 main + relationship tables
- **Records**: ~316 primary records
- **Relationships**: ~400 relationship entries
- **Performance**: ~82s via Local API vs ~6s via SQL (acceptable trade-off)
- **Implementation**: 1-2 days vs 3-5 days for SQL generator

## Collection Dependency Analysis

### Dependency Hierarchy (Execution Order)

```
Level 0 (No Dependencies):
├── users                    # 0 dependencies
├── media                    # 0 dependencies
└── downloads                # 0 dependencies

Level 1 (Depends on Level 0):
├── posts                    # → media (optional), users (optional)
└── courses                  # → downloads[] (hasMany)

Level 2 (Depends on Level 1):
├── course_lessons           # → courses, downloads[] (hasMany)
└── documentation            # → media (optional), downloads[] (hasMany)

Level 3 (Depends on Level 2):
├── course_quizzes           # → courses, course_lessons (optional)
└── surveys                  # → course_lessons (optional), downloads[] (hasMany)

Level 4 (Depends on Level 3):
├── quiz_questions           # → No direct FK, linked via course_quizzes_rels
└── survey_questions         # → No direct FK, linked via surveys_rels
```

**Critical Insight**: Payload uses polymorphic relationship tables instead of direct foreign keys, requiring special handling.

```sql
  title VARCHAR NOT NULL,
  slug VARCHAR NOT NULL UNIQUE,

  downloads_id UUID REFERENCES payload.downloads(id) ON DELETE CASCADE
```

**Seeding Complexity**: Medium

- Relationships: Multiple rows in `courses_rels` for downloads array

```json

}
```

Becomes:
```sql

-- Relationship inserts (handled by Payload Local API)
```

## Payload Relationship Model Deep Dive

### Why Payload Uses *_rels Tables

From Payload's design philosophy:
> "Payload chose not to use typical FK columns for relationships because it would require migrations and additional logic to build queries and join data. This was done to make fewer data migration scenarios when a config changes to use hasMany or not, or when moving to polymorphic relationTo."

### Relationship Table Pattern

**Generic Structure**:
```sql
CREATE TABLE payload.{collection}_rels (
  id SERIAL PRIMARY KEY,
  "order" INTEGER,  -- For array ordering
  parent_id UUID NOT NULL,  -- FK to parent collection
  path VARCHAR NOT NULL,  -- Field name from schema
  
  -- Polymorphic FK columns (one per possible target collection)
  {target_collection}_id UUID,
  ...
  
  FOREIGN KEY (parent_id) REFERENCES payload.{collection}(id) ON DELETE CASCADE,
  FOREIGN KEY ({target}_id) REFERENCES payload.{target}(id) ON DELETE CASCADE,
  
  UNIQUE (parent_id, path, {target1}_id, {target2}_id, ...)
);
```

### Key Properties

1. **Polymorphic**: One table handles all relationship types for a collection
2. **Path-based**: `path` field identifies which schema field the relationship belongs to
3. **Ordered**: `order` field maintains array sequence
4. **Cascade**: ON DELETE CASCADE prevents orphaned records
5. **Unique**: Constraint prevents duplicate relationships

### Example: Course Lesson Relationships

```typescript
// Schema definition
{
  name: 'course_id',
  type: 'relationship',
  relationTo: 'courses'
}
{
  name: 'downloads',
  type: 'relationship',
  relationTo: 'downloads',
  hasMany: true
}
```

```sql
-- Results in:
CREATE TABLE payload.course_lessons_rels (
  id SERIAL PRIMARY KEY,
  "order" INTEGER,
  parent_id UUID NOT NULL,
  path VARCHAR NOT NULL,
  
  courses_id UUID,  -- For course_id field
  downloads_id UUID  -- For downloads[] field
);

-- Single relationship (course_id)
INSERT INTO payload.course_lessons_rels (parent_id, path, courses_id)
VALUES ('lesson-uuid', 'course_id', 'course-uuid');

-- Array relationship (downloads[])
INSERT INTO payload.course_lessons_rels (parent_id, path, downloads_id, "order")
VALUES
  ('lesson-uuid', 'downloads', 'download1-uuid', 1),
  ('lesson-uuid', 'downloads', 'download2-uuid', 2);
```

## Reference Resolution Strategy

### Reference Format

```typescript
type Reference = `{ref:${string}:${string}}`;

// Examples:
'{ref:courses:ddm}'  // Course by slug
'{ref:downloads:slide-templates}'  // Download by identifier
'{ref:media:/path/to/image.png}'  // Media by path
'{ref:users:admin@example.com}'  // User by email
```

### Resolution Process

```typescript
interface ReferenceMap {
  [collection: string]: {
    [identifier: string]: string; // identifier → UUID
  };
}

// Build map as records are created
const idMap = new Map<string, string>();

// On create:
const created = await payload.create({ collection, data });
idMap.set(`${collection}:${identifier}`, created.id);

// On resolve:
function resolveReference(ref: string, map: ReferenceMap): string {
  const match = ref.match(/\{ref:([^:]+):([^}]+)\}/);
  if (!match) return ref;
  
  const [_, collection, identifier] = match;
  return map[collection]?.[identifier] || ref;
}
```

### Resolution Order Requirements

**Critical**: Must resolve references in dependency order to avoid missing IDs.

**Example Failure Case**:
```typescript
// ❌ WRONG ORDER
await seedLessons();  // References courses (not yet created!)
await seedCourses();

// ✅ CORRECT ORDER
await seedCourses();
await seedLessons();  // References now exist
```

## Lexical Content Handling

### Format

Payload uses Lexical editor, storing content as JSONB:

```json
{
  "root": {
    "type": "root",
    "format": "",
    "indent": 0,
    "version": 1,
    "children": [
      {
        "type": "paragraph",
        "format": "",
        "indent": 0,
        "version": 1,
        "children": [
          {
            "type": "text",
            "format": 0,
            "style": "",
            "text": "Paragraph content",
            "version": 1
          }
        ]
      }
    ]
  }
}
```

### Special Components

**Bunny Video Block** (custom):
```json
{
  "type": "bunny-video",
  "videoId": "2620df68-c2a8-4255-986e-24c1d4c1dbf2",
  "libraryId": "264486",
  "children": [{"type": "text", "text": ""}]
}
```

### Database Storage

```sql
-- Stored as JSONB column
content JSONB

-- Example INSERT
INSERT INTO payload.course_lessons (content)
VALUES ('{"root": {...}}'::jsonb);
```

### Seeding Considerations

1. **Conversion**: Markdown → Lexical JSON (via markdown-to-lexical.ts)
2. **Validation**: Must be valid Lexical format
3. **Size**: Large content fields may impact performance
4. **Custom Blocks**: Must handle Bunny video, YouTube, etc.

## Data Integrity Requirements

### Validation Layers

1. **Schema-level** (Payload config):
   - Required fields
   - Unique constraints
   - Min/max values
   - Field types

2. **Database-level** (PostgreSQL):
   - Foreign key constraints
   - Unique indexes
   - NOT NULL constraints
   - CHECK constraints

3. **Application-level** (Payload hooks):
   - Custom validation logic
   - Cross-field validation
   - Business rules

### Critical Constraints

**Must be enforced for seeding**:

```sql
-- Unique slugs (critical for routing)
UNIQUE (slug) ON courses, course_lessons, course_quizzes, surveys

-- Foreign keys (prevent orphaned records)
ON DELETE CASCADE for all relationship tables

-- Required fields
NOT NULL for title, slug, type, etc.

-- Array validation
At least 1 option for quiz questions
At least 2 options for multiple choice survey questions
```

### Validation Strategy

**Local API Approach**: All validation automatic
- Schema validation from Payload config
- Database constraints enforced by PostgreSQL
- Hooks execute business logic
- Clear error messages on failure

**SQL Approach**: Manual validation required
- Must replicate all schema rules
- Risk of constraint violations
- Silent failures or generic errors

## Recommended Approach: Payload Local API

### Decision Rationale

Based on comprehensive analysis:

**Quantitative Factors**:
- Dataset size: 316 records (well within Local API sweet spot)
- Performance: 82s acceptable for development seeding
- Implementation: 1-2 days vs 3-5 days for SQL
- Risk score: 7/35 (low) vs 24/35 (high) for SQL

**Qualitative Factors**:
- Automatic relationship handling (no *_rels table logic needed)
- Built-in validation (schema + database + hooks)
- Maintainability (adapts to schema changes automatically)
- Type safety (full TypeScript support)
- Developer experience (simple API, clear errors)

### Implementation Architecture

```typescript
// apps/payload/src/seed/index.ts

import { getPayload } from 'payload';
import config from '../payload.config';

export async function seedPayload() {
  const payload = await getPayload({ config });
  
  // Reference map for {ref:} resolution
  const idMap = new Map<string, string>();
  
  // Load seed data
  const seedData = {
    users: loadJSON('seed-data/users.json'),
    media: loadJSON('seed-data/media-references.json'),
    downloads: loadJSON('seed-data/download-references.json'),
    courses: loadJSON('seed-data/courses.json'),
    course_lessons: loadJSON('seed-data/course-lessons.json'),
    course_quizzes: loadJSON('seed-data/course-quizzes.json'),
    quiz_questions: loadJSON('seed-data/quiz-questions.json'),
    surveys: loadJSON('seed-data/surveys.json'),
    survey_questions: loadJSON('seed-data/survey-questions.json'),
    posts: loadJSON('seed-data/posts.json'),
    documentation: loadJSON('seed-data/documentation.json'),
  };
  
  // Seed in dependency order
  for (const [collection, data] of Object.entries(seedData)) {
    console.log(`Seeding ${collection}...`);
    
    for (const item of data) {
      // Resolve {ref:} placeholders
      const resolved = resolveReferences(item, idMap);
      
      // Create via Local API
      const created = await payload.create({
        collection: collection,
        data: resolved,
      });
      
      // Store for future reference resolution
      if (item._ref) {
        idMap.set(item._ref, created.id);
      }
    }
  }
}
```

### Key Components

1. **Reference Resolver** (~100 lines):
   - Parse `{ref:collection:identifier}` format
   - Lookup in ID map
   - Replace with actual UUIDs
   - Handle nested references

2. **Data Loader** (~50 lines):
   - Load JSON files
   - Validate structure
   - Handle missing files

3. **Seeder** (~150 lines):
   - Initialize Payload
   - Process collections in order
   - Error handling
   - Progress logging

**Total Code**: ~300 lines vs ~3,000 for SQL generator
