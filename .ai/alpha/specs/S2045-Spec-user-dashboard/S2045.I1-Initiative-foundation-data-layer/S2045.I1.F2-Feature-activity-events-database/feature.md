## Description
Create the `activity_events` table with schema, RLS policies, indexes, and 5 `AFTER INSERT` triggers on source tables (`course_progress`, `lesson_progress`, `quiz_attempts`, `survey_responses`, `building_blocks_submissions`). Generate Supabase TypeScript types. This provides the data foundation for the activity feed widget in I3.

## User Story
**As a** SlideHeroes user
**I want** my actions (lessons completed, quizzes taken, surveys completed, presentations created) to be automatically recorded
**So that** my dashboard activity feed can show a timeline of my learning journey

## Architecture Decision

**Approach**: Pragmatic
**Rationale**: Follow the exact RLS and trigger patterns established in existing migrations. Use `AFTER INSERT` triggers (not `BEFORE`) to avoid blocking source table writes. Store derived `title` text in the event row for fast timeline rendering without JOINs.

### Key Architectural Choices
1. Denormalize `title` into activity_events (e.g., "Completed Lesson 5") so the activity feed never needs to JOIN source tables
2. Use `AFTER INSERT OR UPDATE` on `course_progress` to capture both `course_started` and `course_completed` events
3. Use `metadata JSONB` for event-specific data (quiz score, lesson name, etc.) â€” flexible schema for future event types

### Trade-offs Accepted
- Denormalized titles may become stale if source data changes (acceptable: activity events are historical records)
- 5 triggers add small overhead on writes (mitigated by using `AFTER` not `BEFORE`)
