# Research: @payloadcms/storage-s3 Plugin Without Upload Field Configuration

**Date:** 2025-10-31
**Researcher:** Claude (Research Orchestrator)
**Status:** Complete

## Research Summary

The `@payloadcms/storage-s3` plugin in Payload CMS is **fundamentally designed to work with collections that have `upload` field configuration**. Using it with collections that lack upload configuration (manual URL management approach) creates a critical architectural mismatch that causes validation errors.

## Key Findings

### 1. S3Storage Plugin Requires Upload Configuration

**Critical Discovery:** The s3Storage plugin expects collections to have Payload's `upload` configuration enabled. When attached to a collection, it:

- **Automatically sets `disableLocalStorage: true`** for each configured collection
- **Hooks into Payload's upload lifecycle** (file processing, validation, storage)
- **Expects file upload operations** through Payload's upload system
- **Validates against upload field requirements** including the reserved `filename` field

**Source Evidence:**
- Plugin TypeScript definition shows it extends `@payloadcms/plugin-cloud-storage/types`
- Collections parameter expects `UploadCollectionSlug` type
- GitHub issue #6640 confirms `file` and `filename` are reserved field names in upload collections

### 2. Reserved Field Name Conflicts

**Issue:** Payload CMS has reserved field names that conflict when using manual field definitions:

- **`file`** - Reserved for upload field operations
- **`filename`** - Reserved for upload metadata
- **`mimeType`** - Expected to be auto-populated by upload system
- **`filesize`** - Expected to be auto-populated by upload system
- **`url`** - Generated by storage adapter, not meant for manual input

**When you define these as regular fields** (text, number, etc.) in a collection without upload config, Payload throws validation errors like:
```
ValidationError: The following field is invalid: filename
```

This happens because:
1. The s3Storage plugin is attached to the collection
2. The plugin expects these to be system-managed fields
3. Your manual field definitions conflict with the plugin's expectations

**Source:** GitHub issue #6640 documented this exact conflict and resulted in PR #7242 to document reserved field names.

### 3. Plugin Behavior Without Upload Field

**What happens when s3Storage is attached to a collection without `upload` config:**

1. **Plugin still initializes** and adds its hooks to the collection
2. **Expects upload operations** but collection has no upload capability
3. **Field validation conflicts** occur for any manually defined upload-related fields
4. **URL generation functions** are registered but have no file data to work with
5. **Storage operations fail** because there's no file buffer to upload

**The plugin cannot function in "read-only" or "metadata-only" mode** - it's designed for active file upload management.

### 4. Seeding Pre-Uploaded Files - Official Approaches

**From GitHub discussions and documentation:**

**Approach A: Use Upload Configuration with File Paths (Supported)**
```typescript
// Collection with upload config
export const Media: CollectionConfig = {
  slug: 'media',
  upload: {
    staticDir: './media', // Local path for development
    disableLocalStorage: true, // When using cloud storage
  },
  fields: [
    { name: 'alt', type: 'text', required: true }
  ]
}

// Seeding with actual files
await payload.create({
  collection: 'media',
  data: { alt: 'My image' },
  filePath: '/path/to/pre-uploaded-file.jpg', // Payload processes this
})
```

**Approach B: Direct Database Seeding (Workaround - Not Recommended)**
- Discussed in GitHub issue #693
- Developer tried to create media records without files
- Got `MissingFile: No files were uploaded` error
- Payload team confirmed this approach is not supported
- The upload system **requires actual file operations**

**Approach C: Using Hooks with Manual Upload (Complex)**
- From GitHub discussion #5565
- Upload files manually to S3 first
- Use `beforeChange` hook to validate and store metadata
- Still requires `upload` configuration on collection
- Not suitable for seeding scenarios

### 5. Why Downloads Works But Media Doesn't

**Your observation:** Downloads collection seeds successfully, Media collection fails.

**Explanation:** This suggests an inconsistency in your setup:

1. **Media collection** has s3Storage plugin attached AND manual field definitions = CONFLICT
2. **Downloads collection** either:
   - Has s3Storage plugin NOT attached, OR
   - Has different field configuration, OR
   - Is seeding different data that doesn't trigger validation

**Verification needed:** Check your payload.config.ts to confirm which s3Storage plugins are actually enabled.

### 6. The Architectural Mismatch

**Your Current Approach:**
```typescript
// Collection WITHOUT upload config
export const Media: CollectionConfig = {
  slug: 'media',
  // No upload configuration
  fields: [
    { name: 'filename', type: 'text', required: true }, // ❌ Conflicts with plugin
    { name: 'url', type: 'text', required: true },      // ❌ Should be generated
    { name: 'mimeType', type: 'text' },                 // ❌ Should be auto-set
    { name: 'filesize', type: 'number' },               // ❌ Should be auto-set
  ]
}

// In payload.config.ts
s3Storage({
  collections: {
    media: { /* config */ }  // ❌ Plugin expects upload operations
  }
})
```

**What the Plugin Expects:**
```typescript
// Collection WITH upload config
export const Media: CollectionConfig = {
  slug: 'media',
  upload: {
    staticDir: './uploads/media',
    disableLocalStorage: true, // Plugin handles this
  },
  fields: [
    // System-managed fields (filename, url, mimeType, filesize) added automatically
    { name: 'alt', type: 'text' }, // ✅ Custom fields only
  ]
}
```

## Recommendations

### Option 1: Use Standard Upload Configuration (Recommended)

**Remove manual field definitions and use Payload's upload system:**

```typescript
// Media.ts
export const Media: CollectionConfig = {
  slug: 'media',
  upload: {
    disableLocalStorage: true, // s3Storage plugin handles storage
    imageSizes: [
      { name: 'thumbnail', width: 400, height: 300, position: 'centre' },
      { name: 'card', width: 768, height: 512, position: 'centre' },
    ],
  },
  fields: [
    // System fields (filename, url, mimeType, filesize, etc.) added automatically
    // Only add custom fields:
    { name: 'alt', type: 'text', required: true },
    { name: 'caption', type: 'text' },
    { name: 'type', type: 'select', options: [...] },
    { name: 'tags', type: 'array', fields: [...] },
  ]
}
```

**Seeding with actual files:**
```typescript
import fs from 'fs'

// Option A: Seed with file buffer
const fileBuffer = fs.readFileSync('/path/to/image.jpg')
await payload.create({
  collection: 'media',
  data: { alt: 'Description' },
  file: {
    data: fileBuffer,
    name: 'image.jpg',
    mimetype: 'image/jpeg',
    size: fileBuffer.length
  }
})

// Option B: Seed with file path (if Payload supports)
await payload.create({
  collection: 'media',
  data: { alt: 'Description' },
  filePath: '/path/to/image.jpg'
})
```

**Pros:**
- ✅ Works with s3Storage plugin as designed
- ✅ Automatic field management (filename, url, mimeType, filesize)
- ✅ Image resizing and processing
- ✅ Proper validation
- ✅ Admin UI upload functionality works

**Cons:**
- ❌ Requires processing files during seeding (slower)
- ❌ More complex seeding logic
- ❌ May need to download files from R2 to re-upload

### Option 2: Remove s3Storage Plugin (Current Workaround)

**Keep manual URL management without storage plugin:**

```typescript
// Media.ts - Current configuration (no changes needed)
export const Media: CollectionConfig = {
  slug: 'media',
  // No upload config
  fields: [
    { name: 'filename', type: 'text', required: true },
    { name: 'url', type: 'text', required: true },
    // ... other manual fields
  ]
}

// payload.config.ts - Remove s3Storage for media
plugins: [
  // Remove or disable:
  // s3Storage({ collections: { media: { ... } } })

  // Keep for downloads if it works:
  s3Storage({ collections: { downloads: { ... } } })
]
```

**Pros:**
- ✅ Fast seeding with pre-uploaded files
- ✅ No validation conflicts
- ✅ Simple data structure
- ✅ Works with existing R2 files

**Cons:**
- ❌ No automatic file upload in admin UI
- ❌ Manual URL management required
- ❌ No image processing/resizing
- ❌ No validation of file existence
- ❌ Inconsistent with Payload patterns

### Option 3: Hybrid Approach (Conditional Plugin)

**Use different configurations for development vs seeding:**

```typescript
// payload.config.ts
const isSeeding = process.env.PAYLOAD_SEEDING === 'true'

export default buildPayload({
  collections: [Media, Downloads],
  plugins: [
    // Only enable s3Storage when NOT seeding
    ...(!isSeeding ? [
      s3Storage({
        collections: {
          media: { disableLocalStorage: true }
        },
        bucket: process.env.R2_BUCKET,
        config: { /* ... */ }
      })
    ] : [])
  ]
})

// Media.ts - Conditional upload config
export const Media: CollectionConfig = {
  slug: 'media',
  upload: process.env.PAYLOAD_SEEDING === 'true' ? undefined : {
    disableLocalStorage: true,
  },
  fields: [/* ... */]
}

// Run seeding
// PAYLOAD_SEEDING=true pnpm seed
```

**Pros:**
- ✅ Fast seeding with manual URLs
- ✅ Admin upload works in development
- ✅ Flexible for different environments

**Cons:**
- ❌ Complex configuration
- ❌ Environmental inconsistency
- ❌ May cause cache issues
- ❌ Harder to maintain

## Root Cause Analysis

### Why the Error Occurs

1. **Plugin Attachment**: s3Storage plugin is attached to `media` collection in payload.config.ts
2. **Field Conflict**: Collection defines `filename` as a manual text field (required: true)
3. **Plugin Expectation**: Plugin expects `filename` to be system-managed by upload infrastructure
4. **Validation Failure**: When seeding tries to set `filename` directly, Payload's validation rejects it because:
   - The field is recognized as reserved by the plugin hooks
   - The value should come from file upload operation, not manual input
   - The plugin's validation hooks mark it as invalid

### Why Downloads Might Work

If downloads is seeding successfully, it's likely because:

1. **Different plugin attachment**: downloads s3Storage might be disabled/not configured
2. **Different field names**: downloads might use different field naming
3. **Different seeding data**: downloads seed data might not trigger the same validation
4. **Timing difference**: downloads might seed before plugins fully initialize

**Verification Command:**
```bash
# Check which plugins are actually enabled
grep -A 20 "s3Storage" apps/payload/src/payload.config.ts
```

## Conclusion

**Primary Answer to Your Questions:**

1. **Does s3Storage add/modify field validation?**
   - YES - It hooks into Payload's upload system which validates upload-related fields
   - It expects fields like `filename`, `url`, `mimeType`, `filesize` to be system-managed
   - Manual field definitions conflict with these expectations

2. **What happens when s3Storage is attached without upload field?**
   - ARCHITECTURAL MISMATCH - The plugin cannot function properly
   - Field validation conflicts occur for reserved names
   - Upload operations fail because there's no upload infrastructure
   - The plugin is not designed for this use case

3. **Special requirements/conflicts with manual URL management?**
   - YES - FUNDAMENTAL INCOMPATIBILITY
   - s3Storage expects active file upload operations
   - Manual URL management bypasses the plugin's entire purpose
   - You must choose: use upload system with plugin OR manual URLs without plugin

4. **Recommended approach for seeding pre-uploaded files?**
   - **Best:** Remove s3Storage plugin from media collection (Option 2)
   - **Alternative:** Use upload config and re-process files during seed (Option 1)
   - **Complex:** Conditional configuration for different environments (Option 3)

**Your Specific Issue:**

The error `"The following field is invalid: filename"` occurs because:
- s3Storage plugin is attached to media collection
- Plugin expects `filename` to be system-managed from upload operations
- Your collection defines `filename` as manual text field (required: true)
- Validation rejects direct assignment to reserved field name

**Fix:** Remove s3Storage plugin from media collection configuration, since you're using manual URL management.
