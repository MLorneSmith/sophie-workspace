# Payload CMS Database Migrations

## Overview

We use manual migrations for Payload CMS database schema changes to avoid compatibility issues with PostgreSQL. This document outlines the process for managing these migrations.

## Problem Background

Payload CMS's PostgreSQL adapter attempts to use the PostgreSQL `serial` type for auto-incrementing ID fields. However, our database configuration encounters compatibility issues with this type, resulting in errors like:

```
error: type "serial" does not exist
code: '42704'
```

To solve this issue, we've implemented a manual migration approach that:

1. Disables automatic schema pushing in Payload CMS
2. Uses explicit SQL migrations with PostgreSQL-compatible types
3. Integrates with our existing content migration system

## Migration Process

### Creating a New Migration

When you need to modify the database schema (add a collection, modify fields, etc.):

1. Create a new migration file:

   ```bash
   cd apps/payload
   pnpm run generate:migration
   ```

2. Edit the generated SQL file in `apps/web/supabase/migrations/payload/`.

3. Test your migration:
   ```bash
   pnpm run apply:migrations  # This runs supabase:reset on the web package
   ```

### Running Migrations

We have several commands for managing migrations:

- `pnpm --filter payload run apply:migrations`: Runs database reset (supabase:reset) which applies all migrations
- `pnpm --filter @kit/content-migrations migrate:all`: Runs all content migrations
- `pnpm --filter payload run migrate:complete`: Runs both schema and content migrations in the correct order

### Integration with Content Migrations

The content migration system now includes schema validation to ensure that the required database schema exists before attempting to migrate content. If schema validation fails, you'll be prompted to run database migrations first.

## Best Practices

### SQL Migration Guidelines

- Use `INTEGER GENERATED BY DEFAULT AS IDENTITY` instead of `serial` type
- For relationships between collections, use appropriate references
- Always include `created_at` and `updated_at` timestamps
- Add appropriate indexes for performance
- Test migrations thoroughly before committing

### Collection Naming Conventions

- Use snake_case for collection names (e.g., `course_quizzes`)
- Use singular form for collection names (e.g., `quiz_question` not `quiz_questions`)
- Match field names in TypeScript collections with column names in the database

### Relationship Fields

When defining relationship fields in Payload collections:

```typescript
{
  name: 'quiz',  // This should match the column name in the database
  type: 'relationship',
  relationTo: 'course_quizzes',
  required: true,
}
```

Ensure the column name in the database matches the field name in the collection definition.

## Common Migration Patterns

### Adding a New Collection

```sql
-- Create the main collection table
CREATE TABLE IF NOT EXISTS "payload"."new_collection" (
  "id" varchar PRIMARY KEY NOT NULL,
  "title" varchar NOT NULL,
  "slug" varchar NOT NULL UNIQUE,
  "content" jsonb,
  "created_at" timestamp(3) with time zone DEFAULT now() NOT NULL,
  "updated_at" timestamp(3) with time zone DEFAULT now() NOT NULL
);

-- Create relationship tables if needed
CREATE TABLE IF NOT EXISTS "payload"."new_collection_rels" (
  "id" INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  "parent_id" varchar NOT NULL REFERENCES "payload"."new_collection"("id") ON DELETE CASCADE,
  "path" varchar NOT NULL,
  "order_index" integer,
  "field" varchar NOT NULL,
  "related_collection" varchar NOT NULL,
  "related_id" varchar NOT NULL
);
```

### Modifying an Existing Collection

```sql
-- Add a new column
ALTER TABLE payload.existing_collection
ADD COLUMN new_field VARCHAR;

-- Modify a column
ALTER TABLE payload.existing_collection
ALTER COLUMN existing_field TYPE TEXT;

-- Add a constraint
ALTER TABLE payload.existing_collection
ADD CONSTRAINT unique_field UNIQUE (field_name);
```

## Troubleshooting

### Schema Validation Errors

If content migrations fail with schema validation errors:

1. Check that you've run database migrations first: `pnpm --filter payload run apply:migrations`
2. Verify that collection names in TypeScript match table names in the database
3. Ensure field names in TypeScript match column names in the database

### Migration Errors

If migrations fail:

1. Check the error message for specific issues
2. Verify that your SQL is compatible with PostgreSQL
3. Check for typos in table or column names
4. Ensure that referenced tables exist before creating foreign key constraints

## Development Workflow

The recommended workflow for making changes to Payload collections:

1. Create/modify Payload collections in TypeScript
2. Create a corresponding database migration using our generator
3. Run the database migrations
4. Create/run content migrations as needed

This approach separates concerns clearly:

- Database schema is managed through explicit SQL migrations
- Content is managed through our existing migration tools
